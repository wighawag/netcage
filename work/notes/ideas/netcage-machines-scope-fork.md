# netcage "machines": the scope fork between an egress jail and a light container runtime

Proposed idea, captured 2026-07-10. NOT tasked, NOT decided. This is a design exploration written before any code; it exists so the thinking does not evaporate and so a future PRD/ADR has a grounded starting point. It relates to ADR-0001/0002 (rootless design), ADR-0006 (sidecar-owns-firewall seam), ADR-0009 (kept pair), ADR-0011 (`netcage start` revive), ADR-0013 (host-identity hardening scope), ADR-0017 (uid-scoped graphroot), and the sibling ideas `anonymity-shell-and-lan-split-tunnel.md` and `time-namespace-clock-boot-fingerprint.md`.

It states a scope fork, RECOMMENDS a side of it (as a recommendation to decide from, not a ruling), maps each namespace to the specific host-recon leaks it does and does not close, proposes a persistence model, sketches the verb surface, shows how the new isolation composes with the existing forced-egress guarantee, and states the residual plainly in the ADR-0013 house style ("what this does NOT close"). It does not implement anything, and it does not itself DECIDE anything: the scope-fork decision, if taken, is recorded later as a terse ADR that points back here for the reasoning. The recommendation is deliberately conservative about the identity of the tool, because the honest answer to "how far should a machine go" is itself the thing worth a decision.

## TL;DR: the settled design (read this first)

This note grew through a maintainer conversation (2026-07-10); the original body below explores the fork from scratch and the "Update" sections record the decisions. If you are picking this up cold, here is the settled picture, so you can continue WITHOUT re-reading the whole exploration. Everything here is DECIDED (maintainer-confirmed) unless the "Open decisions" section at the very bottom lists it as open.

- **The scope fork lands at Point B, SPLIT across repos.** "Point B" = close the host-recon leak class (world-readable `/etc/passwd` real names, `/etc/machine-id`, `/etc/hostname`, other users' processes, host open ports, LAN topology) using mount + PID + UTS + user namespaces ON TOP OF netcage's existing forced-egress guarantee. netcage itself does NOT become a machine runtime (that was the rejected "Point C"). Instead the machine experience is a NEW tool that drives netcage.
- **Repos, clean ownership** (three core + one provisioner + the two existing tools):
  - **netcage (this repo)** grows ONLY the jail-intrinsic half of B: the container-create-time `/etc`-identity binds (a minimal `:ro` `/etc/passwd`+`/etc/group` with only the machine's own generic user; a per-machine `/etc/machine-id`; the per-machine `/etc/hostname`/UTS), the LAN-topology `/sys` close, and a `verify` assertion that these binds stay egress-neutral. This is a small, in-character extension of ADR-0013 (same seam as the existing `/etc/hosts :ro` bind). Every consumer (a bare `netcage run`, anon-pi, the new tool) then gets these for free, proven by `verify`.
  - **`anonbox`** (NEW repo, `github.com/wighawag/anonbox`, Go): the tool-agnostic machine manager. Owns the account/home/lifecycle and replicates ANONCTL'S VERB API (`add / rm / list / status / verify / use / exec / seed-home / update`) over netcage. It is anon-pi's machine layer generalized (drop any shell/tool into an anonymized jailed machine, not just pi).
  - **`anoncore`** (NEW repo, `github.com/wighawag/anoncore`, Go): a shared Go module imported by BOTH anonctl and anonbox, holding the security-critical, substrate-agnostic core: account provisioning (no sudo/wheel grant), `seed-home` (setuid-bit strip, symlink refusal), self-elevation UI, and the world-readable credential-free marker contract. Shared (not copied) so the hardening cannot drift between the two tools. The per-tool substrate glue stays per-repo (anonctl: nftables+shim+lanexempt; anonbox: netcage argv).
  - **`anonseed`** (NEW repo, Go; name SETTLED in update 8, was provisionally `anon-provision`): a SHARED, config-seeding tool with MULTIPLE built-in seed types in ONE repo (pi first, opencode etc. next), plus a reserved git-style PATH-plugin escape hatch for third-party seeds. Given a seed type (`anonseed pi ...`) it writes that tool's config into a machine/account's default-home and declares the allow-direct IP exception the tool needs (e.g. a LAN AI server), for BOTH anonctl (writes `/etc/anonctl/default-home/` + `defaults.json`) and anonbox (same, plus provides the container with the tool staged in). Shares `anoncore`'s seed-home/credential-guard. Provisioning-ONLY (no runtime launcher; launching is the `exec` verb's job, update 7).
  - **existing tools:** **anonctl** (unchanged in mission; gains `anoncore` as a dependency, a retroactive founding-ADR note that the core moved out, and an `exec` verb, see below). **anon-pi** is RETIRED as a runtime tool (update 7): its provisioning half becomes the `anonseed` pi seed, and its runtime half is DELETED (not relocated), because personas remove project-scoped boxes and a generic `exec` verb replaces per-tool arg-forwarding. "Run pi anonymized" becomes: provision once (`anonseed pi`), then `anonbox exec <machine> pi ...` / `anonctl exec pi ...`.

**Naming convention (settled, update 8):** the new repos are `anonbox`, `anonseed`, `anoncore`, ALL run-together (no hyphen), matching the one PUBLISHED, permanent anchor `anonctl` (`anon`+common-word suffix, like `systemctl`/`distrobox`). `anon-pi`'s hyphen is NOT a precedent: it hyphenated because `pi` is a separate product name, and it is being retired anyway, so it does not anchor the convention. So: `anonctl` / `anonbox` / `anonseed` / `anoncore`, uniform.
- **A machine = a dedicated host account + one assigned container + a mounted home.**
  - **Account:** `anonbox-<name>` (with bare `anonbox` as the empty-suffix default). This is a THIRD non-colliding account namespace alongside anonctl's `anon`/`anon-<name>` and anon-pi's `anonpi`/`anonpi-<name>` (which already coexist), so all three tools run on one host without collision. The account exists for ON-HOST UNLINKABILITY only (a mode-700 DAC wall; username-free mount-source paths and file ownership), which is what closes the operator-username residual ADR-0013 could only reduce to a numeric uid. It does NOT need nftables/shim/per-UID forcing, because ANONYMITY (egress) is netcage's job, proven by `netcage verify`. So anonbox lifts anoncore's account-hardening but NOT anonctl's egress half.
  - **Container:** each machine owns one long-lived, NAMED, KEPT netcage container (anon-pi's "durable box" lifecycle, i.e. no `--rm`), revived on login via `netcage start` (ADR-0011). A verb REPLACES the container (`rebuild`/`set-image`) from a new image without touching the account or home. The home is the durable identity; the container is resettable convenience state (rootfs/packages).
  - **Home:** mount the machine's mode-700 HOME ONLY into the jail as `$HOME`. anonbox is tool-agnostic and deliberately does NOT adopt anon-pi's per-project `/projects` mount (that is a pi-specific concept, kept in anon-pi's thin layer). Host files are mounted per-run via netcage's own `-v` if wanted.
- **Login gate (anonctl parity):** `anonbox use <name>` runs `netcage verify` (bound to the machine's proxy) and drops the shell ONLY on green; refuses on red, dropping nothing. anonbox's own `verify` DELEGATES the egress proof to `netcage verify` and adds only account-layer assertions (right account, home ownership, `/etc`-identity binds present). It does NOT build a second egress prover.
- **`exec` verb on BOTH tools (update 7, this is what removes the runtime launcher):** `anonctl exec pi <args...>` / `anonbox exec <machine> pi <args...>` runs a program in the anonymized identity, forwarding all args verbatim, behind the SAME verify-green-then-run gate as `use`. Mechanically a small extension of the existing `use` shell-drop (anonctl `use` already does `setpriv ... <shell> -l`; `exec` does `setpriv ... <shell> -lc "<program> ..."` or execs the program; anonbox does `netcage start` + run-inside). This generic verb is why NO pi-specific runtime launcher is needed: launching any tool is `exec <program> ...`, and pi's only surviving footprint is the `anon-provision` pi config plugin.
- **Reboot PRESERVES a machine** (account + home + kept container all survive; netcage's `/var/tmp` graphroot is reboot-safe, ADR-0013/0017). Unlike anonctl there is NO boot-time forcing to re-arm (netcage applies egress fresh on each `start`), so anonbox needs no `anonctl-nftables.service`-style boot unit: same durability promise as anonctl, simpler reboot story.
- **anon-pi is SPLIT** (update 6): its PROVISIONING half (derive `models.json` from the matching local provider, generate `settings.json`, pick the allow-direct LLM host:port, stage pi into the image) moves into the shared `anon-provision` tool as plugin #1; its RUNTIME half (projects-as-cwd, `--session`/`--fork`, pi-arg forwarding, menu) stays a THIN pi launcher over `anonbox use` / `anonctl use`. anon-pi's REQUIREMENTS guide both the anonbox API and the anon-provision plugin design even though those are built first. Note the language split: anon-provision/anonbox/anoncore are Go; any remaining pi runtime launcher drives them as subprocesses over the on-disk contract + CLI (as anon-pi drives netcage today), not as libraries.
- **Build order (update 8b front-loads two pieces):** `anonctl exec` (pure anonctl, extend `use_exec.go`) and `anonseed` (the config provisioner, pi seed targeting anonctl's existing `/etc/anonctl` hooks) are built FIRST, before anonbox, since neither needs anonbox. Then the fuller sequence: on-disk contract SPEC, `anoncore` extraction, `anonbox`, retire anon-pi's runtime. netcage's jail-intrinsic half lands independently (useful to any consumer). See "Next steps" for the concrete order.

## Context: the concrete gap this closes

netcage's one guarantee is forced NETWORK egress: every TCP and DNS packet from a jailed tool is pushed through the socks5h proxy, fail-closed, and `verify` proves it (CONTEXT.md, ADR-0001/0003). ADR-0013 already drew a scope line: netcage hides the CHEAP+OWNED host-identity leaks (host machine name via `/etc/hosts`, operator username via the graphroot path, host NIC name/MAC via pasta interface naming) and states the rest as an explicit non-goal, because a container shares the host kernel and is not a VM.

The motivating incident lives in the sibling repo anonctl and in this repo's own observation `work/notes/observations/jail-leaks-host-identity-metadata-not-network.md`. anonctl forces egress for one Unix account (`meta skuid`), fail-closed, exactly like netcage does for one netns. An agent running in an anonctl anon account leaked NO real IP (egress held, exits were Tor), yet it fingerprinted the host in detail from ordinary unprivileged sources: world-readable `/etc/passwd` (every account's login name plus GECOS real name), `/etc/hostname`, `/etc/machine-id`, `/proc` (other users' processes), local interfaces and neighbours (LAN IP, router, other LAN devices, Tailscale IP), installed packages, open ports. anonctl is UID-only and structurally cannot mask any of that: it never changes the filesystem or process view the anon account sees, only the fate of its packets. netcage, being namespace-based, is the natural place to close the host-recon class, because a netns machine can also get its own mount, PID, UTS, and user namespaces, which is precisely the isolation anonctl cannot reach.

So the question is not "should netcage close host-recon" (it can, where anonctl cannot) but "how much of a machine should netcage grow into while doing it, and at what point does it stop being an egress jail". Note this is a real advantage worth naming: the anonctl finding `learning-from-anonctl-tails-leak-catalogue.md` already records that the netns model is structurally tighter than the per-UID model on the egress axis (no per-UID transition escape). The recon axis is the second place the netns model dominates the UID model, and machines is where netcage would cash that in.

## The scope fork, stated explicitly

A netcage "machine" could sit at one of three points on a spectrum. The fork is where netcage's identity is decided, so it is stated before the recommendation.

**Point A: network-namespace-only (today's netcage, renamed).** A machine is a kept, named, forced-egress jail with a persistent filesystem. This is essentially ADR-0009's kept pair plus ADR-0011's revive plus a name and a home, dressed as a first-class "machine". It closes NONE of the new host-recon leaks: the tool still shares the host's `/proc`, the host's `/etc/passwd` (whatever the base image ships, or the host's if bind-mounted), the host's process table. The egress guarantee is unchanged and fully proven. netcage stays exactly what it is: an egress jail.

**Point B: egress jail plus the four "cheap recon" namespaces (mount + PID + UTS + user), no persistent multi-machine identity beyond that.** A machine adds a masked `/proc`, a minimal bind-mounted `/etc/passwd`, a per-machine hostname (UTS), an isolated process view (PID), and "root in the machine is not root on the host" (user ns). This closes the host-recon CLASS that motivated the whole exercise, while keeping the egress guarantee as the load-bearing contract. Crucially, most of Point B is already what a plain podman container gives you: podman containers already run in their own mount, PID, UTS, and (rootless) user namespaces by default. netcage today does NOT expose that as a hardening surface because it never framed the container as a "machine" with its own identity; it framed it as a jailed tool run. Point B is therefore less "add four namespaces" and more "own, curate, and PROVE the isolation podman already provides, then close the specific recon leaks that leak THROUGH it" (chiefly `/etc/passwd`, `/etc/machine-id`, and any host paths bind-mounted in).

**Point C: a light container runtime with persistent VMs, per-machine users, snapshots, and a login manager.** A machine is a durable named environment you provision once and log into forever, with its own package set, its own home, its own `machine-id`, lifecycle verbs that feel like `lxc`/`incus`/`machinectl`. This is a genuine container-runtime product. The egress guarantee becomes one property of a machine among many, rather than the whole point of the tool.

**The identity-of-the-tool line runs between B and C, not between A and B.** This is the crux of the fork and it deserves to be stated sharply. Point B is still "an egress jail that also hides host recon": the guarantee is still forced egress, the new namespaces are hardening ON TOP of that guarantee (exactly the shape ADR-0013 already established for `/etc/hosts`, the graphroot, and the NIC name). Point C is where netcage becomes "a light container runtime that also does egress": the centre of gravity moves from the packet fate to the machine lifecycle, persistence becomes a first-class promise rather than a convenience, and netcage starts competing with incus/toolbox/distrobox on their turf while carrying an anonymity guarantee they do not. That is a different product with a different maintenance surface (snapshots, GC, per-machine user management, image lifecycle) and a different failure mode (a machine that persists is a machine that accumulates identity and credentials, which is the SELF-DEFEATING trap the `anonymity-shell-and-lan-split-tunnel.md` note already identified: persistence re-imports the very fingerprint the jail exists to shed).

### Recommendation

**Adopt Point B. Draw the ADR line explicitly at B, and treat any move toward C as a separate, later decision that must be argued on its own.**

Rationale:

- Point B closes the motivating host-recon class (the whole reason this exploration exists) without changing netcage's identity. It is the natural continuation of ADR-0013's "close the cheap+owned leaks, state the residual" posture, one namespace-shaped rung up from the file-and-interface fixes ADR-0013 already shipped.
- Point B reuses the machinery netcage already has. A "machine" at Point B is ADR-0009's kept pair plus ADR-0011's revive plus a name plus a home plus a curated set of namespace hardenings. The egress model, the sidecar, the `verify` leak-test, the graphroot relocation (ADR-0013/0017) all carry over unchanged. There is no new isolation MODEL, only a curated exposure of isolation podman already provides plus a few specific recon fixes.
- Point B keeps `verify` load-bearing and unweakened (see the composition section below). The new namespaces are orthogonal to egress; they do not touch the firewall or the TUN path, so the three-point leak-test stays exactly as strong.
- Point C's persistence promise fights the anonymity goal unless it is designed with the credential-shedding constraint front and centre. That is a real design project, not a feature flag, and it should not be smuggled in under "add persistence to machines". Recommending B keeps persistence honest (a machine's home is convenience storage the user OWNS and can wipe, not a promised durable identity) and defers the harder C decision until there is a concrete need that B cannot serve.

The cost of B is that it does more than "egress jail" implies, so the name and the docs must move with it: a "machine" is an egress jail plus a stated, proven set of host-recon hardenings, and the residual (kernel, timing, shared LAN if bridged, host root) must be documented in the ADR-0013 style so the machine's fingerprint is not surprising. That documentation cost is the price of not silently becoming a container runtime. NOTE: the maintainer update below RELOCATES most of the machine concept out of netcage into a new sibling tool, and adds account-per-machine; read the recommendation through that update.

## Update 2026-07-10 (maintainer): the machine concept lives in a NEW sibling tool, not in netcage; and a machine is backed by its own host ACCOUNT

Three decisions from the maintainer that reshape the recommendation without reversing it. They are grounded in what the sibling repos already do (`../anonctl`, `../anon-pi`), read live 2026-07-10.

### 1. Point C is confirmed overblown, and the machine experience is a SEPARATE tool that USES netcage

Agreed: Point C (netcage itself becoming a light container runtime with a machine lifecycle) is overblown and rejected. The machine experience (named persistent machines, a login front door, list/status/rm) is a NEW tool built ON TOP of netcage, exactly as anon-pi is a "thin, opinionated launcher over `netcage run`" that "never touches Podman directly and never weakens the forced-egress invariant" (anon-pi README/CONTEXT). netcage stays the tool-agnostic jail; the new tool holds the machine opinion. This keeps netcage's identity as an egress jail intact (the B/C line holds) while the machine concept lives one layer up where persistence, login, and lifecycle belong.

This is not hypothetical: anon-pi ALREADY IS this tool, but pi-specific. It already has "machines" (an image + a persistent host home bind-mounted at `/root`), `machine add/list/set-image/rm`, per-machine conversations, a menu front door, `forward`/`ports` delegated to netcage, and `--delete-home`. The new tool is the GENERALIZATION of anon-pi's machine layer: same shape, minus the pi-specific opinion (no models.json/local-model/project-as-conversation-key machinery), so it can drop ANY shell/tool into an anonymized jailed machine, not just pi. Concretely, the goal stated by the maintainer is to REPLICATE THE ANONCTL API over netcage: the new tool's verb surface should mirror anonctl's `add / rm / list / status / verify / use / exec / seed-home / update` one-to-one (`exec` added per update 7), so a user who knows anonctl knows this tool, the only difference being the isolation substrate (netcage netns+namespaces instead of anonctl per-UID nftables).

Why anonctl's API specifically: anonctl and this new tool are the SAME product at different isolation strengths. anonctl is per-UID egress forcing that, by its own README's "NOT defended" section, explicitly CANNOT close the host-recon class ("a tool running as the anon account can still read ... world-readable `/etc/passwd` ... closing it needs namespace/VM-strength isolation ... which is a different tool") and NAMES netcage as that different tool. So the new tool is literally the netcage-backed sibling anonctl already points at: same `add/use/verify` ergonomics, same verify-green-then-drop-the-shell safety gate, but the netns+namespace substrate closes the recon class anonctl's per-UID substrate structurally cannot.

### 2. A machine is backed by its own host ACCOUNT (this is what fixes the mount-source username leak, and it is why account-per-machine, not just namespace-per-machine)

The sharpest new requirement: a machine must run under its OWN dedicated host account (like anonctl's `anon-<name>` account per identity, and exactly like anon-pi's "persona": a dedicated Unix account `anon-<name>` with its own mode-700 home, its own netcage store, and its own fail-closed egress). The REASON is a leak the pure-namespace Point B does NOT close: netcage's own ADR-0013/0017 relocate the graphroot to a uid-scoped `/var/tmp/netcage-storage-<uid>` and bind-mount the machine home from a host path, so the mount SOURCE paths in `/proc/self/mountinfo` carry the operator's numeric uid (and any home bind carries the login user's ownership/path). Running the machine under its OWN account makes those uid-scoped paths and file ownerships belong to the MACHINE'S account, not the operator's login user, so the mount table and file ownership stop pointing at the real user. This is the on-host unlinkability half that a namespace alone cannot give: anon-pi's CONTEXT calls it the "DAC discoverability boundary", two personas "unlinkable both on-host (separate DAC walls) and on the network (separate exits)".

This is a genuine addition to the earlier three-point fork: Point B as originally written was namespace-per-machine INSIDE one login user. The account-per-machine model is strictly stronger and is the right target, because it closes the operator-username residual that ADR-0013 could only push from login-name to numeric-uid, and it gives per-machine on-host unlinkability (separate homes behind mode-700 DAC walls) plus the natural place for per-machine egress (each account its own proxy, the anonctl `<account>@` Tor-isolation-username trick anon-pi already uses). It also inherits anonctl's honest caveat: the account boundary is plain Unix DAC, a discoverability + unlinkability boundary against an unprivileged host agent, NOT hard containment against host root (documented, not defended). Provisioning accounts needs root and self-elevation (anonctl's ufw stance, self-elevating verbs); the new tool inherits that model, and anon-pi's `persona add` + self-re-exec via `sudo -u anon-<name> -i` is the proven pattern.

### 3. Therefore: split the work by "whose job is it", and if the new tool can do a piece of B, the new tool does it

The maintainer's framing question ("if some of B can be made by that new tool, that new tool should probably do it, no?") is exactly right, and it gives a clean partition rule: a piece of Point B belongs in NETCAGE only if it is intrinsic to the JAIL (must be applied at container-create time, needs the netns/firewall/sidecar, or is a leak netcage's own construction introduces); everything else belongs in the NEW TOOL. Applying that rule:

- **netcage's job (jail-intrinsic parts of B):** the mount-ns `/etc` binds that must ride the `--network container:` create (a minimal `:ro` `/etc/passwd`/`/etc/group`, a per-machine `/etc/machine-id`, the per-machine `/etc/hostname`/UTS), because these are container-create-time binds on the jailed container netcage already owns (direct analogue of ADR-0013's `/etc/hosts` bind, same seam). The PID/mount/user namespaces (already podman defaults netcage owns). Closing the LAN-topology `/sys` residual in the netns. And `verify` growing an assertion that these binds are egress-neutral. These stay in netcage because they are properties of the jail, and because putting them in netcage means EVERY consumer (anon-pi, the new tool, a bare `netcage run`) gets them for free and proven by `verify`. A likely shape: a netcage flag/mode (e.g. `netcage run --machine-identity <name>` or a small set of `--etc-passwd`/`--machine-id` binds) the new tool composes, mirroring how anon-pi composes `--allow-direct` + `-v home:/root` today.
- **the new tool's job (everything account/lifecycle/persistence):** provisioning the dedicated host account (root, self-elevating, like anonctl `add`), the persistent home directory under that account's tree, the machine metadata/registry, the `add/use/exec/list/status/rm/seed-home/update` verb surface (the anonctl API, `exec` per update 7), the verify-green-then-drop-the-shell login gate, per-machine egress (each account its own proxy), and running netcage AS that account so the store/home land in the account's own uid-scoped paths. None of this needs the netns; all of it needs root and host-account management netcage deliberately does NOT do (netcage is rootless and side-effect-honest, ADR-0013).

Net: adopt B, but SPLIT it. netcage grows only the jail-intrinsic `/etc`-identity binds + the namespace ownership + the LAN `/sys` close + the `verify` assertion (a small, in-character extension of ADR-0013). The account, the persistence, the machine lifecycle, and the anonctl-API verb surface all live in the new tool, which drives netcage the way anon-pi does. The new tool is anon-pi generalized (tool-agnostic) with anonctl's exact verb API. This keeps netcage an egress jail (the B/C line holds and is now cleaner, because the machine LIFECYCLE is provably not in netcage), while the recon class is genuinely closed end-to-end: the jail-intrinsic half by netcage, the account/username half by the new tool's dedicated-account model.

### 4. The new tool is `anonbox`, a Go repo, sharing a Go module with anonctl; anon-pi drives its API; its accounts are `anonbox-<name>` and need NO nftables

Name and build decisions from the maintainer:

- **Name: `anonbox`.** A NEW repo (`github.com/wighawag/anonbox`), family-consistent with `anonctl`/`anon-pi`, reads as "an anonymized machine/box", and does not shadow a `podman`/`netcage` verb (the ADR-0012 naming rule the whole family follows). It generalizes anon-pi's machine layer to be tool-agnostic.
- **Language: Go**, not TypeScript. anon-pi is TS because it lives in the pi/npm world (`npx`, reads `~/.pi/agent/models.json`); the GENERIC tool has none of that, it is OS/exec/account work Go does better, and the account/verify/provision machinery it needs ALREADY EXISTS as tested Go in anonctl (`internal/provision`, `accountconfig`, `seedhome`, `marker`, `verify`, `sudoprobe`, `ui`). So this is less "rewrite from scratch" and more "re-compose anonctl's Go internals against the netcage substrate instead of nftables", plus a rewrite of anon-pi's small (~3 source files, the ~11k-line body is mostly the pi opinion + tests) netcage-driving launcher layer.
- **Shared Go module for the security-critical core.** anonbox and anonctl SHARE a Go module for the provision/verify/marker/seedhome/elevation logic (account creation with no sudo/wheel grant, setuid-bit stripping on seeded files, minimal login PATH, self-elevation, the world-readable credential-free marker contract). This is deliberate because that logic is security-critical and identical in spirit across both tools; copy-and-diverge would let the hardening drift in two places (dangerous). The substrate-specific glue (anonctl: nftables+shim; anonbox: netcage argv) stays per-repo. This shared-vs-copy call is the kind of hard-to-reverse tradeoff that warrants the new repo's FOUNDING ADR.
- **anon-pi GUIDES the anonbox API, then rebuilds elegantly on top.** anon-pi is the reference consumer: anonbox's verb surface + on-disk contract (account naming, home path, machine registry, the marker) must be designed so anon-pi can become a THIN pi-opinion layer over anonbox (models.json seeding, project-as-conversation-key, pi-arg forwarding) and nothing more. So anon-pi's REQUIREMENTS drive the anonbox API design (what does anon-pi need to stop owning accounts/homes/personas and just delegate?), even though anonbox is built first and anon-pi is rewritten onto it afterward. Define the on-disk contract as a SPEC first (the pattern anon-pi already uses to consume anonctl's `/etc/anonctl/<account>.json` marker), so the two repos never disagree on the layout mid-migration.
- **anonbox accounts are `anonbox-<name>` and need NO nftables/shim/per-UID forcing (the key simplification vs anonctl).** The account prefix `anonbox-<name>` (with `anonbox` as the empty-suffix default, mirroring anonctl's `anon`/`anon-<name>` and anon-pi's `anonpi`/`anonpi-<name>`) is a THIRD non-colliding namespace: anonctl and anon-pi already coexist precisely because anon-pi carved out `anonpi` to avoid anonctl's `anon-` (anon-pi even guards that no `anonpi-<name>` can equal `anon`). So `anonbox-<name>` slots in as the third, and all three tools can run on one host without account collision. CRUCIALLY, an anonbox account does NOT need anonctl's per-UID nftables ruleset or its shim: the ANONYMITY comes from netcage (the netns jail forces egress, fail-closed, proven by `netcage verify`), NOT from per-UID kernel forcing. anonbox uses the dedicated account ONLY for on-host unlinkability (the DAC wall, the username-free mount-source paths and file ownership), NOT for egress forcing. This makes anonbox's account model STRICTLY SIMPLER than anonctl's: it lifts anonctl's `provision`/`seedhome`/`ui`/`sudoprobe` hardening (the account-creation + setuid + self-elevation half), but it does NOT lift `nftables`/`forcing`/`shim`/`lanexempt` (the per-UID egress half), because netcage owns egress. The account is the identity/unlinkability boundary; netcage is the egress boundary; they compose, and neither reimplements the other. (This also means anonbox's `verify` DELEGATES the egress proof to `netcage verify` and adds only account-layer assertions: right account, home ownership, the `/etc`-identity binds present. It does not grow a second egress prover.)

### 5. The shared module is its OWN repo; a machine = an account + an ASSIGNED container + a mounted home; reboot preserves (anonctl-API parity); home-only mount (no per-project machine)

Two refinements from the maintainer, both grounded in what anon-pi/anonctl already do.

**5a. The shared security core lives in a SEPARATE third repo, not inside anonctl.** Extract the provision/verify/marker/seedhome/elevation core into its own small Go module (e.g. `github.com/wighawag/anoncore` or similar), which BOTH anonctl and anonbox import. Preferred over hosting it inside the anonctl repo because: hosting-in-anonctl would make anonctl a library as well as a binary and couple anonbox to anonctl's release cadence and internal layout (a refactor in anonctl's `internal/` would ripple into anonbox); a standalone module has a stable import path and its own version, and neither consumer owns the other. The cost is a third repo to maintain and a founding ADR for the module boundary (what is shared core vs per-tool substrate glue: shared = account creation with no sudo/wheel grant, setuid-strip on seed, minimal login PATH, self-elevation, the world-readable credential-free marker contract; per-tool = anonctl's nftables/shim/lanexempt, anonbox's netcage argv). This is the first hard-to-reverse decision, so it is the founding ADR of the shared module repo, referenced by both anonbox's and (retroactively) anonctl's founding ADRs.

**5b. A machine is an account + an ASSIGNED, REPLACEABLE container + a mounted home; reboot PRESERVES (anonctl-API parity); mount the HOME ONLY (no per-project machine).** anon-pi already has both container lifecycles (ADR-0004): a default THROWAWAY container (`--rm`, all valuable state in the durable host home) AND an explicit DURABLE named box (omits `--rm`, revived via `netcage start <name>`). anonbox takes the DURABLE-box model as its DEFAULT, because anonctl-API parity requires it: an anonctl account persists across reboots (its forcing re-arms at boot), so an anonbox machine must feel equally durable. Concretely:

- **Each account is ASSIGNED a container.** A machine (account `anonbox-<name>`) owns one long-lived, named, kept netcage container (no `--rm`), the anon-pi "durable box" shape. Its filesystem state (installed packages, rootfs changes) persists in the kept container; `use`/`enter` is `netcage start <name>` (ADR-0011 revive), verify-green-then-shell.
- **Reboot PRESERVES the machine.** Parity with anonctl: after a reboot the machine still exists (its account, its home, its kept container in netcage's uid-scoped store all survive; netcage's graphroot is `/var/tmp`, reboot-safe, ADR-0013/0017). Unlike anonctl there is NO boot-time forcing to re-arm (netcage owns egress, applied fresh on each `netcage start`), so anonbox needs NO `anonctl-nftables.service`-style early boot unit: reboot-preservation for anonbox is just "the account + home + kept container are still on disk", and the next `use` revives the jail. Simpler reboot story than anonctl, same durability promise.
- **Offer a command to REPLACE the container.** Because the durable identity is the ACCOUNT + HOME, the container is replaceable underneath them: a verb (e.g. `anonbox rebuild <name>` / `set-image`, mirroring anon-pi's `machine set-image`) swaps the machine's assigned container for a fresh one from a new/updated image, WITHOUT touching the account or the home. This is the clean separation the throwaway-vs-durable tension resolves into: the home is always durable (it is the valuable state, bind-mounted in), the container is durable-by-default-but-replaceable (convenience state you can reset). "Reset this machine's rootfs but keep my home/identity" = replace the container; "delete this machine" = remove account + home + container.
- **Mount the HOME ONLY (the deliberate divergence from anon-pi).** anon-pi mounts a per-machine home at `/root` AND a separate per-PROJECT folder at `/projects/<name>` (because pi keys a conversation by cwd, so projects are a pi-specific concept). anonbox is tool-agnostic and needs NEITHER the project concept NOR the project mount: it mounts only the machine's persistent HOME (the account's mode-700 home dir) into the jail as the machine's `$HOME`. No projects root, no `/projects`, no per-project machine. A user who wants to work on host files mounts them explicitly per-run (netcage's own `-v`), exactly as bare `netcage run` already supports; anonbox does not grow a projects abstraction. This keeps anonbox strictly the anonctl API (account + home + verify + use) over netcage, and leaves project/cwd opinions to the consumer (anon-pi keeps its `/projects` concept in its thin layer, since that is pi-specific).

### 6. anon-pi's PROVISIONING half becomes a shared, tool-pluggable Go provisioner (`anon-provision`); its RUNTIME half stays a thin pi launcher (fork flagged open)

Maintainer idea (2026-07-10), a further simplification of Step 4 ("rewrite anon-pi on top of anonbox"). Grounded in what anon-pi's `init` actually does, read live: its PROVISIONING concern is small and self-contained (derive a `models.json` from the local provider whose URL matches the LLM endpoint, generate a `settings.json`, pick the `--allow-direct` LLM `host:port`, choose/build an image with pi staged into `/opt/anon-pi-seed`), while everything else (the launcher menu, projects-as-cwd, `--session`/`--fork`, pi-arg forwarding) is RUNTIME.

The idea: split anon-pi along that seam and make the provisioning half a SEPARATE, SHARED, tool-pluggable Go tool, provisionally `anon-provision`. What it does: given a "tool set" (pi being the FIRST; the design supports more), it writes the config that tool needs into a machine/account's default-home AND declares the IP exception (allow-direct) that tool needs to reach a local service (e.g. a LAN AI server), so the tool is ready to run anonymized. It applies to BOTH substrates:

- **With anonctl:** it writes into anonctl's EXISTING box-wide defaults, `/etc/anonctl/default-home/` (the seed template every fresh account's home is populated from) and `/etc/anonctl/defaults.json` (`allowDirect`). anonctl already has exactly these hooks (README "Seeding a home and box-wide add-time defaults"), and anonctl's README already points at anon-pi as the tool that can generate a pi `models.json` for you, so this is formalising a seam anonctl anticipated.
- **With anonbox:** it does the same home/allow-direct seeding, AND additionally provides/selects the CONTAINER that has the tool staged in (the image with pi on PATH + the `/opt/...-seed` staging), since an anonbox machine is account + container + home. So the anonbox path of the provisioner covers the image half anonctl does not have.

Why this is attractive: the provisioning work is substrate-agnostic file-writing + endpoint-probing + config generation (no pi/npm runtime), so it is naturally Go and can SHARE `anoncore`'s `seed-home`/marker/credential-guard logic (including anon-pi's rule: never seed a real-looking apiKey into an anon home). Making it TOOL-PLUGGABLE (pi is plugin #1: it knows pi's `models.json`/`settings.json` shape and pi's LLM-endpoint convention; a later plugin knows another tool's config + its own direct-exception need) turns "anonymize pi" into one instance of "anonymize any local-service-using tool", which is the same generalisation move anonbox makes for the machine concept.

**The one fork to decide (flagged open, NOT resolved here): provisioning-only vs provisioner-plus-launcher.** Provisioning and RUNTIME are different jobs and pi genuinely needs both. Provisioning = one-shot at `add` time (seed `~/.pi/agent/`, pick allow-direct, stage the image). Runtime = every launch (projects-as-cwd because pi keys conversations by cwd, `--session`/`--fork`, forwarding pi args, the menu). `anon-provision` cleanly owns provisioning; the runtime opinion does not vanish and must live somewhere. Two options:

- **(a) provisioning-only (preferred, truer to the simplification):** `anon-provision` only provisions. The pi RUNTIME launcher becomes a separate thin thing that does `anonbox use <machine>` (or `anonctl use`) then runs pi inside with the right cwd/args. If anonbox drops you into a shell with the home mounted, "run pi in a project" may shrink to `cd <mount> && pi ...`, so the pi runtime launcher could become almost nothing (best case, and the vindication of this whole idea). That thin launcher can stay pi-specific (even a small script or Go helper) or be a plugin's optional "run" entrypoint.
- **(b) provisioner-plus-launcher:** `anon-provision` grows a per-tool "run" entrypoint too, so the pi plugin contributes BOTH how-to-provision-pi AND how-to-launch-pi-in-the-machine. Heavier, keeps one tool, but re-imports the runtime concern into the shared provisioner the split was meant to shed.

Preference is (a): the appeal is that provisioning is the shared, boring, substrate-agnostic part, and the runtime opinion is small and belongs wherever pi's launcher lives, not in the shared provisioner. But this is a real fork with a genuine tradeoff (one-tool simplicity vs clean separation), so it is DECIDED when `anon-provision` is designed, not here. Naming is also provisional (`anon-provision` vs folding this into `anoncore` as a `seed`/`provision` verb set vs another name); pin it at design time. (RESOLVED in updates 7+8: fork -> (a); name -> `anonseed`.)

Net effect on the plan: Step 4 ("rewrite anon-pi on anonbox") splits into (4a) build `anon-provision` (Go, tool-pluggable, pi as plugin #1, seeds home + allow-direct for both anonctl and anonbox, provides the container for anonbox, shares `anoncore`), and (4b) whatever thin pi RUNTIME launcher remains (per the fork above). anon-pi as a standalone TS tool is then either retired or reduced to that thin launcher. UPDATE 7 below RESOLVES the (a)/(b) fork to (a) and removes the runtime launcher entirely via an `exec` verb.

### 7. The (a)/(b) fork is RESOLVED to (a): no runtime launcher at all, because personas remove project-scoped boxes AND both tools gain an `exec <program> ...` verb

Maintainer decision (2026-07-10): option (a), and stronger than "the runtime launcher lives elsewhere", there is NO runtime launcher, because two things remove the need:

- **Personas remove the project-scoped box, which was the launcher's whole job.** anon-pi's runtime launcher exists mostly to manage PROJECT-scoped boxes (a project mounted at `/projects/<name>`, pi keying a conversation by that cwd, `--session`/`--fork` across projects). With the persona/account-per-machine model (updates 2 + 5b, home-only mount, no `/projects`), there ARE no project-scoped boxes to manage: the durable unit is the persona (account + home + container), full stop. So the state the launcher held is gone, not relocated. Nothing needs projects-as-cwd because the machine IS the scope.
- **Both anonctl and anonbox gain an `exec` verb** to launch a program in the identity, e.g. `anonctl exec pi <any pi args...>` / `anonbox exec <machine> pi <any pi args...>`, where everything after the program name is forwarded verbatim to that program. This is what replaces the launcher: instead of a pi-specific launcher that forwards pi args, a GENERIC `exec` runs any program in the anonymized identity and passes its args through. pi is then just `anonbox exec <machine> pi -p "..."` / `anonctl exec pi --session <id>`; no pi-specific runtime code anywhere.

Mechanically `exec` is a SMALL extension of the existing `use` shell-drop, confirmed by reading anonctl's `use_exec.go`: `use` today resolves the account's uid/gid/shell and exec-replaces via `setpriv --reuid <uid> --regid <gid> --init-groups <shell> -l` after a green verify. `exec` is the SAME verify-gate + same drop, but instead of an interactive login shell it runs the given program (e.g. `setpriv ... <shell> -lc "pi \"$@\""`, or exec the program directly with the forwarded argv), still verify-green-THEN-run, still refuse on red. For anonctl the drop is `setpriv` into the anon UID (unchanged mechanism); for anonbox it is `netcage start` the machine's container + run the program inside it (the jail is the identity), again verify-green-first. So `exec` inherits the exact anonctl safety gate `use` already has, for free.

Consequences that SIMPLIFY the whole plan:

- **`anon-provision` is provisioning-only (option a, settled).** It seeds config + declares the allow-direct and (for anonbox) provides the staged container. It carries NO runtime "run" entrypoint. The plugin interface is purely "what config files does this tool need seeded, and what direct-exception does it need", not "how do you launch it" (launching is the generic `exec` verb's job).
- **anon-pi as a runtime tool goes away.** There is no thin pi launcher to keep. "Run pi anonymized" = provision once with the pi plugin (`anon-provision`), then `anonbox exec <machine> pi ...` (or `anonctl exec pi ...`). The pi opinion survives ONLY as the `anon-provision` pi PLUGIN (its config-seeding knowledge), not as any runtime code. anon-pi the repo is retired (or becomes a docs/meta pointer at the provision plugin + the exec pattern).
- **`exec` is a new shared verb worth putting in `anoncore`** (the drop-into-identity-and-run mechanism differs only in substrate: anonctl `setpriv`, anonbox `netcage start`+run), so its verify-gated core can be shared while each tool supplies the substrate drop. It also generalises anonctl's `use` (interactive) and `exec` (one program) as two faces of the same verify-then-enter primitive.

So the runtime launcher is not shrunk, it is DELETED: personas kill the project-box state, and a generic verify-gated `exec <program> ...` on both tools replaces per-tool arg-forwarding. This is the cleanest possible landing of update 6's fork.

### 8. The provisioner is named `anonseed`: ONE repo, MULTIPLE built-in seed types, plus a git-style PATH-plugin door for third parties (NOT a repo per tool); and the build order changes: `anonseed` + `anonctl exec` FIRST, before anonbox

Maintainer decisions (2026-07-10), two parts.

**8a. Name and shape.** The provisioner is `anonseed` (settling update 6's provisional `anon-provision`). Rationale for the name: `anon-provision` collided with `anoncore`, whose account-creation IS "provisioning" (anonctl's `internal/provision`); `anonseed` instead matches the vocabulary that already exists (anonctl's `seed-home` verb, `/etc/anonctl/default-home/`) and does not overclaim, it seeds tool CONFIG, it does not provision accounts. Shape: ONE repo with MULTIPLE built-in seed types (`anonseed pi ...`, later `anonseed opencode ...`), NOT a repo per tool (`anonseed-pi`, `anonseed-opencode`, ... REJECTED). Why one repo beats repo-per-tool:
- The shared surface (resolve the target identity; write files into a home safely via `anoncore`'s setuid-strip/symlink-refusal/mode-700 seed-home; declare an allow-direct; enforce the credential-shedding guard) is the BULK of the work and is IDENTICAL across seed types. Repo-per-tool either duplicates it (drift, the exact danger `anoncore` sharing avoids) or makes every tiny tool repo depend on the same libs anyway.
- A seed type is SMALL: mostly config knowledge (pi: probe `/v1/models`, merge with the matching `models.json` provider, generate `settings.json`, refuse a real apiKey, the endpoint convention), a plugin's worth of code, not a repo's worth. A whole repo (README/CI/release/versioning/install) per small config unit is huge overhead for the payload.
- It matches the family (anonctl/netcage are each one binary with subcommands), and one binary can `anonseed list` its known seeds (discovery), where repo-per-tool scatters discovery across GitHub.

**"Ability for others to provide more" WITHOUT repo-per-tool: the git/kubectl PATH-plugin pattern.** `anonseed` ships the common seeds BUILT IN; for an unknown seed, `anonseed foo` falls back to exec-ing a PATH executable `anonseed-foo`. So a third party writes their own `anonseed-<x>` without touching the `anonseed` repo, and YOUR seeds still do not fragment into per-tool repos. Built-in for what you ship; PATH-plugin for the long tail. IMPORTANT: do NOT build the plugin-discovery mechanism NOW, it is speculative until a third-party seed exists; RESERVE it in the design and ship pi as a plain built-in subcommand. (A seed type is pi-specific CODE, not pure data, because pi's merge/probe/refuse logic is non-trivial, so a purely-declarative external-template model is rejected; built-in code + PATH-plugin is the extensible shape.)

**8b. Build order: `anonseed` + `anonctl exec` come FIRST, before anonbox** (maintainer wants these now). Both are buildable without anonbox:
- **`anonctl exec`** is pure anonctl: extend the existing `use_exec.go` setpriv drop to RUN a program (not just an interactive login shell), add `exec` to the verb table, keep the verify-green-then-run gate. Self-contained; no anonbox, no anoncore needed (build it IN anonctl now; when `anoncore` is later extracted, the shared enter-primitive moves there with the rest, do NOT create `anoncore` early just to host one verb).
- **`anonseed`** is substrate-agnostic and only needs anonctl's ALREADY-SHIPPED hooks (`/etc/anonctl/default-home/` + `defaults.json`); the anonbox path (provide/stage the container) is added when anonbox exists. So `anonseed` with the pi seed targeting anonctl is buildable now.
- Note the earlier `anoncore` sequencing (Step 2) is NOT a prerequisite for THIS slice: `anonctl exec` extends anonctl in place, and `anonseed` can vendor/copy the small seed-home/credential-guard helper now and switch to importing `anoncore` when it is extracted (or `anoncore` is extracted first if convenient). The founding `anoncore`/anonbox work still happens later per Steps 2/3; this update only front-loads `anonctl exec` + `anonseed`.

BLOCKER noted at decision time: anonctl had an uncommitted `README.md` change; per the repo's clean-tree rule, `anonctl exec` work must not start until that is committed/stashed. netcage and anon-pi were clean.

The sections below (namespace-to-leak table, persistence, verbs, residual) are written from the earlier "netcage grows the machine" framing. Re-read them with this split in mind: the namespace-to-leak MECHANICS are unchanged (a machine still needs those binds and namespaces), but WHERE each lives moves per the partition rule above, the persistence model becomes the new tool's account-home (not netcage's kept-container store alone), and the verb surface becomes the new tool's (anonctl API), not new `netcage machines` verbs. Rather than rewrite them, they stand as the mechanism reference; this update is the authority on ownership.

This is the concrete mapping. Each row names the leak from the motivating incident, the namespace (or non-namespace mechanism) that closes it, and whether it is CLOSED, PARTIAL, or OPEN at Point B. "Already-by-podman" marks isolation a rootless podman container already provides today, which netcage would OWN and PROVE as a machine property rather than newly build.

### What remains OPEN even at Point B (the shared-kernel residual, unchanged from ADR-0013)

- **Kernel version and hardware** (`uname`, `/proc/version`, `/proc/cpuinfo`, `/proc/meminfo`): OPEN. The kernel is shared; a machine is not a VM. Identical to ADR-0013's accepted residual. No namespace closes this; only a sandbox kernel (gVisor/Kata) or a real VM would.
- **Host clock / boot-time** (`/proc/uptime`, `/proc/stat` `btime`, `/proc/loadavg`): OPEN at Point B unless a time namespace is added. `CLONE_NEWTIME` (the idea in `time-namespace-clock-boot-fingerprint.md`) would give a machine its own boot-time origin and close the `uptime`/`btime` correlator. This is the ONE additional namespace worth considering as a Point-B-plus option, because it is narrow, kernel-supported, and closes a strong stable correlator (`btime` is the machine-id-grade signal of the clock family). It is left as a follow-on to the core B recommendation, not a blocker: the open questions in that idea note (does rootless podman/crun expose the knob, does it compose with the `--network container:` shared-netns sidecar) must be spiked first.

## The persistence model (the crux)

anonctl's "machine" is just a UID, so persistence is free: the account's `$HOME` already exists on the real filesystem and survives every login with zero machinery. A netcage machine has no such gift. Its filesystem is a container rootfs that, by default, is either ephemeral (`--rm`) or a kept container's writable layer that podman owns. To be a machine you can "log into at any time" with your work intact, it needs a persistent home/rootfs that survives across logins, a defined place on disk, and defined teardown/GC and delete semantics. This is the hardest piece and is treated as such.

### The rootfs and its writable layer

A machine is built on an OVERLAY: a read-only lower layer (the base image the machine was created from, e.g. the pinned dev image of ADR-0004 or a user-chosen image) and a writable UPPER layer that captures everything the machine's owner changes (their home, installed packages, edited files). podman already builds exactly this overlay for every container; the design question is where the upper layer lives and whether it survives a container being removed.

Two candidate mechanisms, both reusing existing netcage machinery:

**Option 1: the kept-container writable layer (reuse ADR-0009 + ADR-0011 directly).** A machine IS a kept, named container. Its writable upper layer is the container's own diff layer inside the graphroot (ADR-0013/0017: `/var/tmp/netcage-storage-<uid>/`). "Log in at any time" is `netcage start <machine>` (ADR-0011): revive the sidecar, re-apply the baked firewall, verify green, re-enter the tool with its filesystem state intact. This is the SMALLEST design: a machine is a kept container with a friendlier name and a login front door. Persistence is exactly the kept-container promise (survives reboots, survives between logins, accumulates until cleared). The teardown/GC/delete semantics are already defined: `netcage rm` removes the pair, a graphroot wipe self-heals to empty (destroying the machine's home, exactly as losing the home store would, per ADR-0013).

- Pro: zero new persistence machinery, rides ADR-0009/0011 verbatim, delete is `netcage rm`, the store is already username-free (ADR-0013/0017).
- Con: the machine's home is coupled to podman's container storage lifecycle. A `podman system reset` or a `/var/tmp` tmpfiles-age wipe silently destroys machines (the ADR-0013 kept-run caveat, now sharper because a "machine" implies more durability than a "kept run"). The upper layer is inside podman's graphroot, so the user cannot trivially back it up or move it without `podman unshare` (the id-mapped-subuid `rm -rf` caveat from ADR-0013).

**Option 2: an explicit per-machine home volume on a known path, bind-mounted at login (a thin persistence layer netcage owns).** The machine's rootfs is still an overlay (base image lower + a throwaway or kept upper), but the machine's HOME (`/home/machine`, and optionally a small set of dot-dirs) is a named volume or a plain host directory under an XDG-ish machines root (e.g. `${XDG_DATA_HOME}/netcage/machines/<name>/home`, or a `/var/tmp` sibling to stay username-free like the graphroot). At login netcage bind-mounts that directory to `/home/machine` inside the machine. The container itself can even be ephemeral (`--rm`): the DURABLE part is the home volume, not the container.

- Pro: the machine's home is a plain directory the user OWNS, can back up, can inspect, can wipe, decoupled from podman's storage lifecycle. Reboots and graphroot wipes do not destroy it (it is not in the graphroot). "Delete a machine" is unambiguous: remove that directory (plus any per-machine metadata). It matches the anonctl mental model most closely: the home is a real, persistent, user-owned thing, and the "machine" is the thin identity wrapped around it.
- Con: the home is bind-mounted from a host path, so its SOURCE path is visible in the machine's `/proc/self/mountinfo` (the ADR-0013 `-v` source-path caveat). Keep the machines root username-free (under `/var/tmp/netcage-machines-<uid>/` or an XDG path with a numeric uid, never `/home/<you>/`) so the mount source does not re-leak the operator username the graphroot relocation just fixed. Also: only the HOME persists, not arbitrary rootfs changes (installed packages in `/usr` are in the throwaway upper and are lost unless the machine is a kept container too). That may be fine (a machine's durable state is its home and its work, not its `apt install`s), or it may need Option 1 UNDER Option 2 (kept container for the rootfs, explicit home volume for the durable home).

### Recommendation for persistence

**Start with Option 1 (a machine is a kept container, persistence = the kept-container promise), and offer Option 2's explicit user-owned home directory as the durable-home story layered on top.** Concretely: a machine is a kept, named container (Option 1) whose HOME is a bind-mounted user-owned directory under a username-free machines root (Option 2). The kept container carries the rootfs/package state; the explicit home directory carries the durable, backup-able, unambiguously-deletable home. This gives:

- "Log in at any time" = `netcage start <machine>` (ADR-0011 revive), verify green, drop the shell, with both the rootfs state and the home intact.
- "Delete a machine" = remove the kept pair (`netcage rm`) AND remove the machine's home directory AND its metadata. Two clear on-disk things, both user-owned, both username-free. A machine delete is complete only when both are gone; the verb must do both (or refuse partially and say so).
- Teardown/GC: the container follows ADR-0009 (kept, fail-closed at rest via the baked firewall). The home directory follows nothing automatic: it persists until an explicit delete, exactly like a real home. No auto-cleanup (the ADR-0013 rule: netcage does not auto-destroy storage, because auto-delete silently destroys the state the persistence was for). A `netcage machines gc` could offer to reap machines with no kept container left (orphaned homes) or homes with no machine metadata, but only as an explicit, confirming verb, never a default.

The credential-shedding constraint from `anonymity-shell-and-lan-split-tunnel.md` lands HERE and is load-bearing: a persistent machine home is a place credentials accumulate (`~/.config/gh`, tokens, ssh keys), and a machine that carries credentials stops the IP leak while still authenticating as you, which is self-defeating for the anonymity goal. So the persistence design must NOT default to importing any host config into the machine home. The machine home starts EMPTY (or seeded only with explicitly credential-free scaffolding). Any host-config reuse is the separate, curated, credential-free opt-in that note already scoped, not a machines default.

## Ergonomics parity with anonctl

anonctl's verbs are add / use / list / status / verify. The netcage machines surface should mirror them one-to-one where the semantics match, and diverge only where the netns/rootfs model genuinely differs from the UID model. netcage already owns a `netcage-<verb>` namespace and the ADR-0012 rule that a netcage-only verb must never shadow a podman verb, so a `machines` sub-namespace (`netcage machines <verb>`, or `netcage machine <verb>`) keeps these clearly netcage-owned and away from the podman-faithful pass-through verbs.

| anonctl verb | netcage machines verb | Meaning |
| --- | --- | --- |
| `add` | `netcage machines add <name> [--image <ref>] [--proxy ...]` | Create a named, persistent machine: allocate its home directory (username-free machines root), record its metadata (base image, the proxy/allowlist it is bound to, its per-machine synthetic machine-id and hostname), but do NOT necessarily start it. Idempotent-ish: refuse to clobber an existing machine of the same name. |
| `use` | `netcage machines use <name>` | The front door. Revive the machine's jail (ADR-0011 `start`), APPLY the host-recon hardenings (minimal `/etc/passwd`, per-machine hostname/machine-id, the PID/mount/user namespaces), run `verify` against the machine's bound proxy, and ONLY on green drop the operator into a shell in the machine (`-it`). On RED, refuse and drop NO shell (the anonctl safety gate, preserved exactly). |
| `list` | `netcage machines list [--json]` | List machines by name with status, bound proxy source, base image, last-login. Scoped to netcage's `netcage.managed` label plus a `netcage.machine=<name>` label, mirroring the ADR-0009/0016 pattern; `--json` emits a stable reuse contract like `ports --json` / `detect-proxy --json`. |
| `status` | `netcage machines status <name>` | Report one machine's state: is the jail running, is egress currently green (or when it was last verified), where its home lives on disk, which proxy it is bound to, which hardenings are applied. Read-only, no egress of its own beyond an optional verify. |
| `verify` | `netcage machines verify <name>` (or `netcage verify` bound to the machine's proxy) | Run the existing three-point leak-test through the machine's jail, proving egress is still forced+fail-closed with all the added namespaces in place. This is the same `verify`, pointed at the machine's bound proxy; the machines layer must not weaken it (see composition). |
| (anonctl has no explicit rm; a UID is deleted out of band) | `netcage machines rm <name>` | Delete the machine: remove the kept pair (ADR-0009 `rm` semantics) AND remove the machine's home directory AND its metadata. Complete-or-refuse: a partial delete (container gone, home orphaned) is reported, not silently left. No auto-delete anywhere else guarantees this is the only path that destroys a machine's home. |

### How "log in at any time" works mechanically

The choice is between nsenter-into-a-long-lived-jail versus recreate-and-attach. netcage's existing model already answers this and the machines layer should not diverge:

- netcage does NOT keep a long-lived jail process running between logins. A kept machine is at REST as a STOPPED container pair, held fail-closed by its baked firewall (ADR-0009). There is no long-lived jail to `nsenter` into, and deliberately so: a running-at-rest jail is a bigger attack surface and the "kept pair rest state" observation already flagged that keeping the sidecar running is undesirable.
- So "log in at any time" is RECREATE-AND-ATTACH, specifically the ADR-0011 `netcage start` revive: bring the stopped pair back up, re-apply and re-VERIFY the baked firewall, re-exec the DNS forwarder, and re-enter the tool with its filesystem state intact. `netcage machines use` is `netcage start` with the machine's bound proxy plus the host-recon hardenings plus the verify-then-shell gate.
- The safety gate is preserved byte-for-byte from anonctl: verify GREEN, THEN drop the shell; on RED, refuse and drop nothing. ADR-0011 already verifies the revived jail before re-entry; the machines `use` verb makes that verify the explicit precondition for the login shell, and refuses (non-zero, no shell) on any failed assertion. This is the same posture as `netcage verify` gating CI, applied to an interactive front door.
- `netcage exec` into a machine (ADR-0009's `exec`) still works for a non-login command in an already-running machine, entering the EXISTING jailed netns (never a fresh un-jailed one), consistent with today's behaviour.

## Interaction with the existing egress guarantee

The load-bearing property is that the new namespaces are ORTHOGONAL to egress and therefore do not weaken the proof. Concretely:

- The forced-egress mechanism is unchanged: the machine's traffic still leaves ONLY through the tun2socks sidecar (ADR-0001), the sidecar still owns the baked firewall + DNS forwarder (ADR-0006/0008), UDP/ICMP/non-53 are still hard-dropped (ADR-0003), fail-closed still holds. None of mount/PID/UTS/user (or a future time) namespace touches the firewall, the TUN, the pasta uplink, or the DNS forwarder. They change what the tool can READ about the host (files, process table, hostname, ids), not where its PACKETS can go.
- `verify` stays the acceptance floor and is run through the machine's jail unchanged. The three assertions (exit IP is the proxy's; a unique hostname resolves through the proxy's resolver; proxy-killed means fail-closed) are all about packet fate, which the new namespaces do not affect. So the machines layer inherits the full existing leak-test and its fixture-backed extensions (IPv6 fails closed, non-53 UDP dropped, ICMP dropped, loopback confined, `--allow-direct` is never a clear-DNS hole) at zero cost and with zero weakening.
- The one place the namespaces INTERSECT egress is the minimal `/etc/passwd` / `/etc/hostname` / `/etc/machine-id` binds. These must follow the ADR-0013 `/etc/hosts` precedent exactly: synthesized files, bind-mounted `:ro`, accepted under the `--network container:` topology (ADR-0013 live-verified that `/etc/hosts :ro` and `--hostname` are accepted there, unlike `--dns`). They must NOT reintroduce a DNS path or a hostname->IP pin that sidesteps proxy-side DNS (the reason `--add-host` is refused). A synthetic `/etc/passwd`/`machine-id` carries no name resolution, so it is inert to egress; the machines wiring test should assert egress-neutrality the same way the host-identity-hardening wiring asserted the `/etc/hosts` mount did not affect the three-point test.
- The user namespace is already present under rootless podman (ADR-0001/0002), and the pasta host-loopback reachback is a rootless-userns mechanism (ADR-0002). So "add a user ns" is not a new composition risk; it is the existing one, stated. A machine must not request any capability/device/privilege that the run-flag allowlist (ADR-0010) refuses; the machines verbs inherit that fail-closed allowlist so a machine cannot be provisioned with a jail-breaching flag.

## Honest residual: what a netcage machine does NOT close

Stated plainly, in the ADR-0013 house style, so the machine's fingerprint is documented rather than surprising. Even with all of mount + PID + UTS + user namespaces (and even with a time namespace added), a netcage machine still leaks:

- **Kernel and hardware.** `uname`, `/proc/version`, `/proc/cpuinfo`, `/proc/meminfo` report the shared host kernel and physical CPUs/RAM. A machine is not a VM; no namespace changes this. Closing it needs a sandbox kernel (gVisor/Kata) or a real VM, which is a different isolation model and out of scope for Point B (the same line ADR-0013 already drew).
- **Host clock / boot-time**, UNLESS a time namespace is added. Without `CLONE_NEWTIME`, `/proc/uptime`, `/proc/stat` `btime` (the strong stable correlator), and `/proc/loadavg` reflect the host. Even WITH a time namespace, wall clock (`CLOCK_REALTIME`) and loadavg are not fully virtualised (open questions in the time-namespace idea note).
- **Timing and side channels.** Shared CPU, cache, memory pressure, and scheduling are observable across the namespace boundary. A machine can be fingerprinted or correlated by timing (co-resident load, cache effects, clock skew) regardless of the file-and-process masking. This is inherent to sharing a kernel and hardware and is not something a container-level tool closes.
- **Shared LAN, if the netns still bridges to it.** If a machine's netns carries the host LAN addresses/routes (the pasta interface's copied `192.168.x` addresses, the smaller follow-up ADR-0013 flagged), a tool could still read LAN topology via `/sys` even though its egress route is the TUN. A machine should CLOSE this deliberately (verify no host LAN address is readable in the machine), but until it does, it is an open residual, distinct from the egress guarantee (egress is still forced through the proxy; this is a READ leak of LAN identity, not a packet escaping). `--allow-direct` (ADR-0005), when used, is by definition a named LAN hole and is a deliberate, guardrailed exception, not a residual.
- **Host root and the host itself.** A machine runs on the host; the HOST's root can see everything about the machine (its processes, its files, its memory). The machine protects the operator from the JAILED TOOL fingerprinting the host, not the host operator from the tool via host root, and not the tool from a hostile host. netcage is not a defence against a compromised host.
- **Operator-chosen leaks.** Anything the operator bind-mounts in (`-v` source paths, a shared home, host config) can re-import identity or credentials. The ADR-0013 `-v` source-path caveat and the `anonymity-shell-and-lan-split-tunnel.md` credential-shedding constraint both apply: a machine that mounts the host's `$HOME` or `/etc/passwd` re-leaks exactly what the machine exists to hide. The default machine imports NOTHING; any import is a louder, curated, credential-free opt-in.
- **Image-baked identity.** Whatever the machine's base image ships in its own `/etc` and env (ADR-0013's wrapper/tool-image env residual) is the image's concern, not netcage's. A machine built from an image that bakes an identifying hostname or env inherits it.

## Considered alternatives (why not the neighbours)

- **Point A only (rename kept runs "machines", close no recon).** REJECTED as the whole answer: it ignores the motivating gap. The reason to build machines at all is to close the host-recon class anonctl structurally cannot; Point A does not do that. (Point A's machinery is, however, the FOUNDATION of Point B, so it is not wasted.)
- **Point C (netcage ITSELF becoming a light container runtime with durable per-machine identity and a login manager).** REJECTED (confirmed by the maintainer, see the 2026-07-10 update): it moves netcage's centre of gravity off the egress guarantee, competes with incus/toolbox/distrobox, and its persistence promise fights the anonymity goal unless designed with credential-shedding as a first-class constraint. The machine lifecycle instead lives in a SEPARATE tool that drives netcage (like anon-pi does), so netcage stays an egress jail and the B/C line is enforced by the tool boundary itself, not just by discipline.
- **A VM per machine (real hardware/kernel isolation via Kata/microVM).** REJECTED for Point B: it would close the kernel/hardware/timing residual, but it is a wholesale runtime swap (the same "different isolation model" ADR-0013 puts out of scope), it breaks the rootless-podman design that is load-bearing (ADR-0001/0002), and it is far past the egress-jail identity. Worth naming as the only thing that closes the shared-kernel residual, and therefore the honest ceiling on what a container-based machine can ever hide.
- **A time namespace as part of core Point B.** DEFERRED, not rejected: it is the one extra namespace worth adding after B lands, because it closes a strong stable correlator (`btime`) narrowly and at the kernel level. It is held out of the core recommendation only because its rootless-podman/crun feasibility and its composition with the shared-netns sidecar are unspiked (the open questions in `time-namespace-clock-boot-fingerprint.md`).

## Residual and honest caveats (carry into every tool's docs)

The residual is documented in the ADR-0013 style (see the "Honest residual" section above): kernel/hardware and clock/boot-time (shared kernel, not a VM), timing/side channels, shared LAN if the netns bridges it, host root, and operator-chosen or image-baked leaks. Plus the account-layer residual anonctl already states: the `anonbox-<name>` account boundary is plain Unix DAC, a discoverability + on-host-unlinkability boundary against an UNPRIVILEGED host agent, NOT hard containment against host root (documented, not defended). The single biggest honest caveat: a machine is not a VM, so the shared-kernel/hardware/timing fingerprint is a permanent residual of the container model. anonbox and anon-pi should each carry a "what this does NOT close" section in the netcage/anonctl house style.
