# netcage "machines": the scope fork between an egress jail and a light container runtime

Proposed idea, captured 2026-07-10. NOT tasked, NOT decided. This is a design exploration written before any code; it exists so the thinking does not evaporate and so a future PRD/ADR has a grounded starting point. It relates to ADR-0001/0002 (rootless design), ADR-0006 (sidecar-owns-firewall seam), ADR-0009 (kept pair), ADR-0011 (`netcage start` revive), ADR-0013 (host-identity hardening scope), ADR-0017 (uid-scoped graphroot), and the sibling ideas `anonymity-shell-and-lan-split-tunnel.md` and `time-namespace-clock-boot-fingerprint.md`.

It states a scope fork, RECOMMENDS a side of it (as a recommendation to decide from, not a ruling), maps each namespace to the specific host-recon leaks it does and does not close, proposes a persistence model, sketches the verb surface, shows how the new isolation composes with the existing forced-egress guarantee, and states the residual plainly in the ADR-0013 house style ("what this does NOT close"). It does not implement anything, and it does not itself DECIDE anything: the scope-fork decision, if taken, is recorded later as a terse ADR that points back here for the reasoning. The recommendation is deliberately conservative about the identity of the tool, because the honest answer to "how far should a machine go" is itself the thing worth a decision.

## Context: the concrete gap this closes

netcage's one guarantee is forced NETWORK egress: every TCP and DNS packet from a jailed tool is pushed through the socks5h proxy, fail-closed, and `verify` proves it (CONTEXT.md, ADR-0001/0003). ADR-0013 already drew a scope line: netcage hides the CHEAP+OWNED host-identity leaks (host machine name via `/etc/hosts`, operator username via the graphroot path, host NIC name/MAC via pasta interface naming) and states the rest as an explicit non-goal, because a container shares the host kernel and is not a VM.

The motivating incident lives in the sibling repo anonctl and in this repo's own observation `work/notes/observations/jail-leaks-host-identity-metadata-not-network.md`. anonctl forces egress for one Unix account (`meta skuid`), fail-closed, exactly like netcage does for one netns. An agent running in an anonctl anon account leaked NO real IP (egress held, exits were Tor), yet it fingerprinted the host in detail from ordinary unprivileged sources: world-readable `/etc/passwd` (every account's login name plus GECOS real name), `/etc/hostname`, `/etc/machine-id`, `/proc` (other users' processes), local interfaces and neighbours (LAN IP, router, other LAN devices, Tailscale IP), installed packages, open ports. anonctl is UID-only and structurally cannot mask any of that: it never changes the filesystem or process view the anon account sees, only the fate of its packets. netcage, being namespace-based, is the natural place to close the host-recon class, because a netns machine can also get its own mount, PID, UTS, and user namespaces, which is precisely the isolation anonctl cannot reach.

So the question is not "should netcage close host-recon" (it can, where anonctl cannot) but "how much of a machine should netcage grow into while doing it, and at what point does it stop being an egress jail". Note this is a real advantage worth naming: the anonctl finding `learning-from-anonctl-tails-leak-catalogue.md` already records that the netns model is structurally tighter than the per-UID model on the egress axis (no per-UID transition escape). The recon axis is the second place the netns model dominates the UID model, and machines is where netcage would cash that in.

## The scope fork, stated explicitly

A netcage "machine" could sit at one of three points on a spectrum. The fork is where netcage's identity is decided, so it is stated before the recommendation.

**Point A: network-namespace-only (today's netcage, renamed).** A machine is a kept, named, forced-egress jail with a persistent filesystem. This is essentially ADR-0009's kept pair plus ADR-0011's revive plus a name and a home, dressed as a first-class "machine". It closes NONE of the new host-recon leaks: the tool still shares the host's `/proc`, the host's `/etc/passwd` (whatever the base image ships, or the host's if bind-mounted), the host's process table. The egress guarantee is unchanged and fully proven. netcage stays exactly what it is: an egress jail.

**Point B: egress jail plus the four "cheap recon" namespaces (mount + PID + UTS + user), no persistent multi-machine identity beyond that.** A machine adds a masked `/proc`, a minimal bind-mounted `/etc/passwd`, a per-machine hostname (UTS), an isolated process view (PID), and "root in the machine is not root on the host" (user ns). This closes the host-recon CLASS that motivated the whole exercise, while keeping the egress guarantee as the load-bearing contract. Crucially, most of Point B is already what a plain podman container gives you: podman containers already run in their own mount, PID, UTS, and (rootless) user namespaces by default. netcage today does NOT expose that as a hardening surface because it never framed the container as a "machine" with its own identity; it framed it as a jailed tool run. Point B is therefore less "add four namespaces" and more "own, curate, and PROVE the isolation podman already provides, then close the specific recon leaks that leak THROUGH it" (chiefly `/etc/passwd`, `/etc/machine-id`, and any host paths bind-mounted in).

**Point C: a light container runtime with persistent VMs, per-machine users, snapshots, and a login manager.** A machine is a durable named environment you provision once and log into forever, with its own package set, its own home, its own `machine-id`, lifecycle verbs that feel like `lxc`/`incus`/`machinectl`. This is a genuine container-runtime product. The egress guarantee becomes one property of a machine among many, rather than the whole point of the tool.

**The identity-of-the-tool line runs between B and C, not between A and B.** This is the crux of the fork and it deserves to be stated sharply. Point B is still "an egress jail that also hides host recon": the guarantee is still forced egress, the new namespaces are hardening ON TOP of that guarantee (exactly the shape ADR-0013 already established for `/etc/hosts`, the graphroot, and the NIC name). Point C is where netcage becomes "a light container runtime that also does egress": the centre of gravity moves from the packet fate to the machine lifecycle, persistence becomes a first-class promise rather than a convenience, and netcage starts competing with incus/toolbox/distrobox on their turf while carrying an anonymity guarantee they do not. That is a different product with a different maintenance surface (snapshots, GC, per-machine user management, image lifecycle) and a different failure mode (a machine that persists is a machine that accumulates identity and credentials, which is the SELF-DEFEATING trap the `anonymity-shell-and-lan-split-tunnel.md` note already identified: persistence re-imports the very fingerprint the jail exists to shed).

### Recommendation

**Adopt Point B. Draw the ADR line explicitly at B, and treat any move toward C as a separate, later decision that must be argued on its own.**

Rationale:

- Point B closes the motivating host-recon class (the whole reason this exploration exists) without changing netcage's identity. It is the natural continuation of ADR-0013's "close the cheap+owned leaks, state the residual" posture, one namespace-shaped rung up from the file-and-interface fixes ADR-0013 already shipped.
- Point B reuses the machinery netcage already has. A "machine" at Point B is ADR-0009's kept pair plus ADR-0011's revive plus a name plus a home plus a curated set of namespace hardenings. The egress model, the sidecar, the `verify` leak-test, the graphroot relocation (ADR-0013/0017) all carry over unchanged. There is no new isolation MODEL, only a curated exposure of isolation podman already provides plus a few specific recon fixes.
- Point B keeps `verify` load-bearing and unweakened (see the composition section below). The new namespaces are orthogonal to egress; they do not touch the firewall or the TUN path, so the three-point leak-test stays exactly as strong.
- Point C's persistence promise fights the anonymity goal unless it is designed with the credential-shedding constraint front and centre. That is a real design project, not a feature flag, and it should not be smuggled in under "add persistence to machines". Recommending B keeps persistence honest (a machine's home is convenience storage the user OWNS and can wipe, not a promised durable identity) and defers the harder C decision until there is a concrete need that B cannot serve.

The cost of B is that it does more than "egress jail" implies, so the name and the docs must move with it: a "machine" is an egress jail plus a stated, proven set of host-recon hardenings, and the residual (kernel, timing, shared LAN if bridged, host root) must be documented in the ADR-0013 style so the machine's fingerprint is not surprising. That documentation cost is the price of not silently becoming a container runtime.

## What each namespace buys against the specific leaks

This is the concrete mapping. Each row names the leak from the motivating incident, the namespace (or non-namespace mechanism) that closes it, and whether it is CLOSED, PARTIAL, or OPEN at Point B. "Already-by-podman" marks isolation a rootless podman container already provides today, which netcage would OWN and PROVE as a machine property rather than newly build.

| Host-recon leak (from the incident) | Mechanism | Verdict at Point B | Notes |
| --- | --- | --- | --- |
| Other users' login names + GECOS real names (`/etc/passwd` world-readable) | mount ns: bind a minimal `/etc/passwd` (`:ro`) containing only the machine's own generic user (e.g. `machine:x:1000:1000::/home/machine:/bin/sh`) plus `root`; same for `/etc/group`, `/etc/shadow` absent | CLOSED | This is the sharpest new win and the direct analogue of ADR-0013's `/etc/hosts` fix (same pattern: synthesize a neutral file, bind `:ro`, mirroring the `resolvConfPath` -> `/etc/resolv.conf:ro` precedent). A stock base image already ships its own `/etc/passwd`, so the leak only occurs if the HOST's `/etc/passwd` is bind-mounted in; the fix is "never bind the host's, always bind a synthetic minimal one". |
| Host machine name (`/etc/hostname`, `hostname`) | UTS ns: per-machine hostname set to a neutral fixed value | CLOSED | ADR-0013 already sets a fixed neutral `--hostname` on the tool container; a machine formalises this as per-machine (each machine can carry its own stable neutral name, or all share one). `/etc/hostname` inside the machine is the base image's, not the host's, once the host's is not bind-mounted. |
| `/etc/machine-id` (stable host correlator) | mount ns: bind an absent or per-machine synthetic `machine-id` (`:ro`, or an empty bind, or a per-machine random UUID persisted with the machine) | CLOSED | `machine-id` is a strong stable correlator (like `btime` in ADR-0013). Options: absent (some tools tolerate this), a per-machine random id generated once and persisted in the machine's home (stable within a machine, unlinkable to the host), or the systemd "first boot" empty value. Recommend per-machine random UUID persisted with the machine: stable enough not to be a tell, unlinkable to the host. |
| Other users' processes (`/proc` process table) | PID ns: the machine sees only its own processes; PID 1 is the machine's init/shell | CLOSED (for OTHER users' processes) | already-by-podman: a podman container already has its own PID namespace, so `ps aux` inside sees only the container's processes, not the host's user processes. The machine framing OWNS and states this. Caveat: `/proc` still exposes the shared-kernel clock/hardware family (see OPEN rows), so PID ns closes the process-LIST leak, not the whole `/proc` fingerprint. |
| Masked `/proc` and `/sys` host paths | mount ns: podman's default masked-paths + a fresh `proc` mount in the PID ns | PARTIAL | already-by-podman: podman masks a set of `/proc` paths (`/proc/kcore`, `/proc/keys`, parts of `/sys`) by default. This does NOT mask `/proc/cpuinfo`, `/proc/meminfo`, `/proc/version`, `/proc/uptime`, `/proc/stat`: those are the shared-kernel residual ADR-0013 already accepts. |
| "root in the machine" being root on the host | user ns: the machine's root maps to an unprivileged host subuid | CLOSED (already-by-podman, rootless) | already-by-podman: rootless podman ALREADY runs every container in a user namespace where in-container root is a host subuid, not host root. This is not new isolation to build; it is a property the machine framing states. It matters for the recon story because it means a machine can safely present a convincing "you are root on this box" view (own `/etc/passwd`, own hostname) without that being a host-privilege escalation. |
| Installed host packages (dpkg/rpm db, `/usr` contents) | mount ns: the machine's rootfs is the base image's, not the host's `/` | CLOSED | already-by-podman: a container's `/` is the image, so the host's package db is simply not present. A machine reads ITS image's package set, which is a property of the image the user chose, not the host. (If the user bind-mounts host `/usr` or the host package db, that is their choice, same caveat as `-v` source paths in ADR-0013.) |
| Open ports on the host (`/proc/net/tcp`, listening sockets) | netns (already netcage's core) + PID/mount ns | CLOSED (for HOST ports) | The machine already has its own netns (netcage's core), so `/proc/net/tcp*` shows only the machine's own sockets, never the host's listeners. netcage's own `ports` verb (ADR-0015) reads exactly this in-netns `/proc/net/tcp*`. So host open ports are already invisible to a jailed tool today; the machine framing states it. |
| Local interfaces / LAN IP / router / other LAN devices / Tailscale IP | netns + the existing egress model | PARTIAL, see OPEN rows | The machine's netns does NOT carry the host's LAN interfaces in the full tun2socks jail: the tool's route out is the TUN (ADR-0001), and `CLONE_MAIN=0` stops the pasta host routes cloning into the TUN table (see the NIC investigation in the observation). BUT the pasta interface still exists in the sidecar's netns and copies the host LAN addresses/routes; whether those are readable from the tool side via `/sys` is the smaller follow-up ADR-0013 flagged. A machine should CLOSE this deliberately (see the LAN-topology residual below), not inherit it silently. Tailscale IP specifically leaks only if the host Tailscale interface is visible in the machine's netns; in the TUN jail it is not, because the machine has its own netns with only `lo` + the TUN + the pasta uplink. |

### What remains OPEN even at Point B (the shared-kernel residual, unchanged from ADR-0013)

- **Kernel version and hardware** (`uname`, `/proc/version`, `/proc/cpuinfo`, `/proc/meminfo`): OPEN. The kernel is shared; a machine is not a VM. Identical to ADR-0013's accepted residual. No namespace closes this; only a sandbox kernel (gVisor/Kata) or a real VM would.
- **Host clock / boot-time** (`/proc/uptime`, `/proc/stat` `btime`, `/proc/loadavg`): OPEN at Point B unless a time namespace is added. `CLONE_NEWTIME` (the idea in `time-namespace-clock-boot-fingerprint.md`) would give a machine its own boot-time origin and close the `uptime`/`btime` correlator. This is the ONE additional namespace worth considering as a Point-B-plus option, because it is narrow, kernel-supported, and closes a strong stable correlator (`btime` is the machine-id-grade signal of the clock family). It is left as a follow-on to the core B recommendation, not a blocker: the open questions in that idea note (does rootless podman/crun expose the knob, does it compose with the `--network container:` shared-netns sidecar) must be spiked first.

## The persistence model (the crux)

anonctl's "machine" is just a UID, so persistence is free: the account's `$HOME` already exists on the real filesystem and survives every login with zero machinery. A netcage machine has no such gift. Its filesystem is a container rootfs that, by default, is either ephemeral (`--rm`) or a kept container's writable layer that podman owns. To be a machine you can "log into at any time" with your work intact, it needs a persistent home/rootfs that survives across logins, a defined place on disk, and defined teardown/GC and delete semantics. This is the hardest piece and is treated as such.

### The rootfs and its writable layer

A machine is built on an OVERLAY: a read-only lower layer (the base image the machine was created from, e.g. the pinned dev image of ADR-0004 or a user-chosen image) and a writable UPPER layer that captures everything the machine's owner changes (their home, installed packages, edited files). podman already builds exactly this overlay for every container; the design question is where the upper layer lives and whether it survives a container being removed.

Two candidate mechanisms, both reusing existing netcage machinery:

**Option 1: the kept-container writable layer (reuse ADR-0009 + ADR-0011 directly).** A machine IS a kept, named container. Its writable upper layer is the container's own diff layer inside the graphroot (ADR-0013/0017: `/var/tmp/netcage-storage-<uid>/`). "Log in at any time" is `netcage start <machine>` (ADR-0011): revive the sidecar, re-apply the baked firewall, verify green, re-enter the tool with its filesystem state intact. This is the SMALLEST design: a machine is a kept container with a friendlier name and a login front door. Persistence is exactly the kept-container promise (survives reboots, survives between logins, accumulates until cleared). The teardown/GC/delete semantics are already defined: `netcage rm` removes the pair, a graphroot wipe self-heals to empty (destroying the machine's home, exactly as losing the home store would, per ADR-0013).

- Pro: zero new persistence machinery, rides ADR-0009/0011 verbatim, delete is `netcage rm`, the store is already username-free (ADR-0013/0017).
- Con: the machine's home is coupled to podman's container storage lifecycle. A `podman system reset` or a `/var/tmp` tmpfiles-age wipe silently destroys machines (the ADR-0013 kept-run caveat, now sharper because a "machine" implies more durability than a "kept run"). The upper layer is inside podman's graphroot, so the user cannot trivially back it up or move it without `podman unshare` (the id-mapped-subuid `rm -rf` caveat from ADR-0013).

**Option 2: an explicit per-machine home volume on a known path, bind-mounted at login (a thin persistence layer netcage owns).** The machine's rootfs is still an overlay (base image lower + a throwaway or kept upper), but the machine's HOME (`/home/machine`, and optionally a small set of dot-dirs) is a named volume or a plain host directory under an XDG-ish machines root (e.g. `${XDG_DATA_HOME}/netcage/machines/<name>/home`, or a `/var/tmp` sibling to stay username-free like the graphroot). At login netcage bind-mounts that directory to `/home/machine` inside the machine. The container itself can even be ephemeral (`--rm`): the DURABLE part is the home volume, not the container.

- Pro: the machine's home is a plain directory the user OWNS, can back up, can inspect, can wipe, decoupled from podman's storage lifecycle. Reboots and graphroot wipes do not destroy it (it is not in the graphroot). "Delete a machine" is unambiguous: remove that directory (plus any per-machine metadata). It matches the anonctl mental model most closely: the home is a real, persistent, user-owned thing, and the "machine" is the thin identity wrapped around it.
- Con: the home is bind-mounted from a host path, so its SOURCE path is visible in the machine's `/proc/self/mountinfo` (the ADR-0013 `-v` source-path caveat). Keep the machines root username-free (under `/var/tmp/netcage-machines-<uid>/` or an XDG path with a numeric uid, never `/home/<you>/`) so the mount source does not re-leak the operator username the graphroot relocation just fixed. Also: only the HOME persists, not arbitrary rootfs changes (installed packages in `/usr` are in the throwaway upper and are lost unless the machine is a kept container too). That may be fine (a machine's durable state is its home and its work, not its `apt install`s), or it may need Option 1 UNDER Option 2 (kept container for the rootfs, explicit home volume for the durable home).

### Recommendation for persistence

**Start with Option 1 (a machine is a kept container, persistence = the kept-container promise), and offer Option 2's explicit user-owned home directory as the durable-home story layered on top.** Concretely: a machine is a kept, named container (Option 1) whose HOME is a bind-mounted user-owned directory under a username-free machines root (Option 2). The kept container carries the rootfs/package state; the explicit home directory carries the durable, backup-able, unambiguously-deletable home. This gives:

- "Log in at any time" = `netcage start <machine>` (ADR-0011 revive), verify green, drop the shell, with both the rootfs state and the home intact.
- "Delete a machine" = remove the kept pair (`netcage rm`) AND remove the machine's home directory AND its metadata. Two clear on-disk things, both user-owned, both username-free. A machine delete is complete only when both are gone; the verb must do both (or refuse partially and say so).
- Teardown/GC: the container follows ADR-0009 (kept, fail-closed at rest via the baked firewall). The home directory follows nothing automatic: it persists until an explicit delete, exactly like a real home. No auto-cleanup (the ADR-0013 rule: netcage does not auto-destroy storage, because auto-delete silently destroys the state the persistence was for). A `netcage machines gc` could offer to reap machines with no kept container left (orphaned homes) or homes with no machine metadata, but only as an explicit, confirming verb, never a default.

The credential-shedding constraint from `anonymity-shell-and-lan-split-tunnel.md` lands HERE and is load-bearing: a persistent machine home is a place credentials accumulate (`~/.config/gh`, tokens, ssh keys), and a machine that carries credentials stops the IP leak while still authenticating as you, which is self-defeating for the anonymity goal. So the persistence design must NOT default to importing any host config into the machine home. The machine home starts EMPTY (or seeded only with explicitly credential-free scaffolding). Any host-config reuse is the separate, curated, credential-free opt-in that note already scoped, not a machines default.

## Ergonomics parity with anonctl

anonctl's verbs are add / use / list / status / verify. The netcage machines surface should mirror them one-to-one where the semantics match, and diverge only where the netns/rootfs model genuinely differs from the UID model. netcage already owns a `netcage-<verb>` namespace and the ADR-0012 rule that a netcage-only verb must never shadow a podman verb, so a `machines` sub-namespace (`netcage machines <verb>`, or `netcage machine <verb>`) keeps these clearly netcage-owned and away from the podman-faithful pass-through verbs.

| anonctl verb | netcage machines verb | Meaning |
| --- | --- | --- |
| `add` | `netcage machines add <name> [--image <ref>] [--proxy ...]` | Create a named, persistent machine: allocate its home directory (username-free machines root), record its metadata (base image, the proxy/allowlist it is bound to, its per-machine synthetic machine-id and hostname), but do NOT necessarily start it. Idempotent-ish: refuse to clobber an existing machine of the same name. |
| `use` | `netcage machines use <name>` | The front door. Revive the machine's jail (ADR-0011 `start`), APPLY the host-recon hardenings (minimal `/etc/passwd`, per-machine hostname/machine-id, the PID/mount/user namespaces), run `verify` against the machine's bound proxy, and ONLY on green drop the operator into a shell in the machine (`-it`). On RED, refuse and drop NO shell (the anonctl safety gate, preserved exactly). |
| `list` | `netcage machines list [--json]` | List machines by name with status, bound proxy source, base image, last-login. Scoped to netcage's `netcage.managed` label plus a `netcage.machine=<name>` label, mirroring the ADR-0009/0016 pattern; `--json` emits a stable reuse contract like `ports --json` / `detect-proxy --json`. |
| `status` | `netcage machines status <name>` | Report one machine's state: is the jail running, is egress currently green (or when it was last verified), where its home lives on disk, which proxy it is bound to, which hardenings are applied. Read-only, no egress of its own beyond an optional verify. |
| `verify` | `netcage machines verify <name>` (or `netcage verify` bound to the machine's proxy) | Run the existing three-point leak-test through the machine's jail, proving egress is still forced+fail-closed with all the added namespaces in place. This is the same `verify`, pointed at the machine's bound proxy; the machines layer must not weaken it (see composition). |
| (anonctl has no explicit rm; a UID is deleted out of band) | `netcage machines rm <name>` | Delete the machine: remove the kept pair (ADR-0009 `rm` semantics) AND remove the machine's home directory AND its metadata. Complete-or-refuse: a partial delete (container gone, home orphaned) is reported, not silently left. No auto-delete anywhere else guarantees this is the only path that destroys a machine's home. |

### How "log in at any time" works mechanically

The choice is between nsenter-into-a-long-lived-jail versus recreate-and-attach. netcage's existing model already answers this and the machines layer should not diverge:

- netcage does NOT keep a long-lived jail process running between logins. A kept machine is at REST as a STOPPED container pair, held fail-closed by its baked firewall (ADR-0009). There is no long-lived jail to `nsenter` into, and deliberately so: a running-at-rest jail is a bigger attack surface and the "kept pair rest state" observation already flagged that keeping the sidecar running is undesirable.
- So "log in at any time" is RECREATE-AND-ATTACH, specifically the ADR-0011 `netcage start` revive: bring the stopped pair back up, re-apply and re-VERIFY the baked firewall, re-exec the DNS forwarder, and re-enter the tool with its filesystem state intact. `netcage machines use` is `netcage start` with the machine's bound proxy plus the host-recon hardenings plus the verify-then-shell gate.
- The safety gate is preserved byte-for-byte from anonctl: verify GREEN, THEN drop the shell; on RED, refuse and drop nothing. ADR-0011 already verifies the revived jail before re-entry; the machines `use` verb makes that verify the explicit precondition for the login shell, and refuses (non-zero, no shell) on any failed assertion. This is the same posture as `netcage verify` gating CI, applied to an interactive front door.
- `netcage exec` into a machine (ADR-0009's `exec`) still works for a non-login command in an already-running machine, entering the EXISTING jailed netns (never a fresh un-jailed one), consistent with today's behaviour.

## Interaction with the existing egress guarantee

The load-bearing property is that the new namespaces are ORTHOGONAL to egress and therefore do not weaken the proof. Concretely:

- The forced-egress mechanism is unchanged: the machine's traffic still leaves ONLY through the tun2socks sidecar (ADR-0001), the sidecar still owns the baked firewall + DNS forwarder (ADR-0006/0008), UDP/ICMP/non-53 are still hard-dropped (ADR-0003), fail-closed still holds. None of mount/PID/UTS/user (or a future time) namespace touches the firewall, the TUN, the pasta uplink, or the DNS forwarder. They change what the tool can READ about the host (files, process table, hostname, ids), not where its PACKETS can go.
- `verify` stays the acceptance floor and is run through the machine's jail unchanged. The three assertions (exit IP is the proxy's; a unique hostname resolves through the proxy's resolver; proxy-killed means fail-closed) are all about packet fate, which the new namespaces do not affect. So the machines layer inherits the full existing leak-test and its fixture-backed extensions (IPv6 fails closed, non-53 UDP dropped, ICMP dropped, loopback confined, `--allow-direct` is never a clear-DNS hole) at zero cost and with zero weakening.
- The one place the namespaces INTERSECT egress is the minimal `/etc/passwd` / `/etc/hostname` / `/etc/machine-id` binds. These must follow the ADR-0013 `/etc/hosts` precedent exactly: synthesized files, bind-mounted `:ro`, accepted under the `--network container:` topology (ADR-0013 live-verified that `/etc/hosts :ro` and `--hostname` are accepted there, unlike `--dns`). They must NOT reintroduce a DNS path or a hostname->IP pin that sidesteps proxy-side DNS (the reason `--add-host` is refused). A synthetic `/etc/passwd`/`machine-id` carries no name resolution, so it is inert to egress; the machines wiring test should assert egress-neutrality the same way the host-identity-hardening wiring asserted the `/etc/hosts` mount did not affect the three-point test.
- The user namespace is already present under rootless podman (ADR-0001/0002), and the pasta host-loopback reachback is a rootless-userns mechanism (ADR-0002). So "add a user ns" is not a new composition risk; it is the existing one, stated. A machine must not request any capability/device/privilege that the run-flag allowlist (ADR-0010) refuses; the machines verbs inherit that fail-closed allowlist so a machine cannot be provisioned with a jail-breaching flag.

## Honest residual: what a netcage machine does NOT close

Stated plainly, in the ADR-0013 house style, so the machine's fingerprint is documented rather than surprising. Even with all of mount + PID + UTS + user namespaces (and even with a time namespace added), a netcage machine still leaks:

- **Kernel and hardware.** `uname`, `/proc/version`, `/proc/cpuinfo`, `/proc/meminfo` report the shared host kernel and physical CPUs/RAM. A machine is not a VM; no namespace changes this. Closing it needs a sandbox kernel (gVisor/Kata) or a real VM, which is a different isolation model and out of scope for Point B (the same line ADR-0013 already drew).
- **Host clock / boot-time**, UNLESS a time namespace is added. Without `CLONE_NEWTIME`, `/proc/uptime`, `/proc/stat` `btime` (the strong stable correlator), and `/proc/loadavg` reflect the host. Even WITH a time namespace, wall clock (`CLOCK_REALTIME`) and loadavg are not fully virtualised (open questions in the time-namespace idea note).
- **Timing and side channels.** Shared CPU, cache, memory pressure, and scheduling are observable across the namespace boundary. A machine can be fingerprinted or correlated by timing (co-resident load, cache effects, clock skew) regardless of the file-and-process masking. This is inherent to sharing a kernel and hardware and is not something a container-level tool closes.
- **Shared LAN, if the netns still bridges to it.** If a machine's netns carries the host LAN addresses/routes (the pasta interface's copied `192.168.x` addresses, the smaller follow-up ADR-0013 flagged), a tool could still read LAN topology via `/sys` even though its egress route is the TUN. A machine should CLOSE this deliberately (verify no host LAN address is readable in the machine), but until it does, it is an open residual, distinct from the egress guarantee (egress is still forced through the proxy; this is a READ leak of LAN identity, not a packet escaping). `--allow-direct` (ADR-0005), when used, is by definition a named LAN hole and is a deliberate, guardrailed exception, not a residual.
- **Host root and the host itself.** A machine runs on the host; the HOST's root can see everything about the machine (its processes, its files, its memory). The machine protects the operator from the JAILED TOOL fingerprinting the host, not the host operator from the tool via host root, and not the tool from a hostile host. netcage is not a defence against a compromised host.
- **Operator-chosen leaks.** Anything the operator bind-mounts in (`-v` source paths, a shared home, host config) can re-import identity or credentials. The ADR-0013 `-v` source-path caveat and the `anonymity-shell-and-lan-split-tunnel.md` credential-shedding constraint both apply: a machine that mounts the host's `$HOME` or `/etc/passwd` re-leaks exactly what the machine exists to hide. The default machine imports NOTHING; any import is a louder, curated, credential-free opt-in.
- **Image-baked identity.** Whatever the machine's base image ships in its own `/etc` and env (ADR-0013's wrapper/tool-image env residual) is the image's concern, not netcage's. A machine built from an image that bakes an identifying hostname or env inherits it.

## Considered alternatives (why not the neighbours)

- **Point A only (rename kept runs "machines", close no recon).** REJECTED as the whole answer: it ignores the motivating gap. The reason to build machines at all is to close the host-recon class anonctl structurally cannot; Point A does not do that. (Point A's machinery is, however, the FOUNDATION of Point B, so it is not wasted.)
- **Point C (a light container runtime with durable per-machine identity and a login manager).** REJECTED for now: it moves netcage's centre of gravity off the egress guarantee, competes with incus/toolbox/distrobox, and its persistence promise fights the anonymity goal unless designed with credential-shedding as a first-class constraint. It is a separate product decision that must be argued on its own merits, not reached by accretion from "add persistence to machines".
- **A VM per machine (real hardware/kernel isolation via Kata/microVM).** REJECTED for Point B: it would close the kernel/hardware/timing residual, but it is a wholesale runtime swap (the same "different isolation model" ADR-0013 puts out of scope), it breaks the rootless-podman design that is load-bearing (ADR-0001/0002), and it is far past the egress-jail identity. Worth naming as the only thing that closes the shared-kernel residual, and therefore the honest ceiling on what a container-based machine can ever hide.
- **A time namespace as part of core Point B.** DEFERRED, not rejected: it is the one extra namespace worth adding after B lands, because it closes a strong stable correlator (`btime`) narrowly and at the kernel level. It is held out of the core recommendation only because its rootless-podman/crun feasibility and its composition with the shared-netns sidecar are unspiked (the open questions in `time-namespace-clock-boot-fingerprint.md`).

## What a decision would need to nail down (if this fork is taken)

- The identity line is proposed at Point B: netcage machines are an egress jail plus a stated, proven set of host-recon hardenings. If that is adopted, the ADR that records it must say so in one line and must say that any future move toward Point C (durable per-machine identity, a login manager, snapshots, per-machine users beyond the single generic user) is a SEPARATE later decision that supersedes or extends it explicitly, not one reached by accretion.
- The egress guarantee and `verify` are unchanged and unweakened; the machines layer inherits them wholesale. A machines wiring test must assert egress-neutrality of the new binds, exactly as the host-identity-hardening work asserted for `/etc/hosts`.
- Persistence is a machine's kept-container state plus a user-owned, username-free home directory. Delete means removing BOTH; there is no auto-cleanup; the credential-shedding constraint means the default machine imports no host config.
- The residual is documented in the ADR-0013 style (kernel, timing, shared LAN if bridged, host root, operator-chosen and image-baked leaks), so a machine's fingerprint is stated, not surprising. The single biggest honest caveat: a machine is not a VM, so the shared-kernel/hardware/timing fingerprint is a permanent residual of the container model.
- This idea is INCUBATING, not tasked. The next step, if the fork is taken, is a short ADR recording the Point-B decision (pointing back here for the reasoning), then a PRD under `work/prds/` that turns Point B into slices (the `/etc/passwd` + `machine-id` binds mirroring the ADR-0013 `/etc/hosts` work; the machines verb surface mirroring anonctl; the persistence home-directory story), each gated by the existing `verify` floor, plus a spike for the deferred time namespace. No ADR is written until the fork is decided; no code is written until that PRD exists.
