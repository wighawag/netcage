# The graphroot DEFAULT is UID-scoped (`/var/tmp/netcage-storage-<uid>`) to fix a multi-user-host store collision; `NETCAGE_GRAPHROOT` is promoted to a supported optional override

**Status:** accepted (fixes a latent bug in ADR-0013's fixed `/var/tmp` graphroot; builds on its `--root` single-injection-seam)

## Context

ADR-0013 relocated podman's graphroot to a FIXED literal `/var/tmp/netcage-storage` to close Leak 2: rootless podman's default graphroot is `~/.local/share/containers/storage`, so the overlay lowerdir/upperdir SOURCE paths in the container's `/proc/self/mountinfo` embed `/home/<user>`, leaking the operator's login NAME to the in-jail tool. A username-free `/var/tmp` path makes those mountinfo paths name-free. The relocation rides podman's global `--root` flag, injected at the single `ExecRunner.Run` exec seam so every podman invocation shares ONE store. The path is selected by a pure resolver (`internal/jail/graphroot.go`, `graphRoot()`), which honours a `NETCAGE_GRAPHROOT` env override (documented in code as TEST-ONLY: it exists so an integration test can isolate real storage under a scratch dir), else the fixed default.

**The fixed literal is a latent multi-user bug.** `/var/tmp/netcage-storage` is ONE path shared by every user on the host. netcage never `mkdir`s it: podman creates it lazily on first use (that is the "no provisioning helper, self-healing" property `/var/tmp` was chosen for). So the FIRST Unix user to run netcage creates `/var/tmp/netcage-storage` OWNED BY THAT USER. When a SECOND, different Unix user runs netcage on the same host, they hit a store directory owned by the first user, and rootless podman's per-user subuid-owned overlay tree cannot cohabit one path across two users: the second user collides (permission failure, or a tangled cross-user store). `/var/tmp`'s sticky bit protects only the TOP level (users cannot delete each other's entries); it does NOT make the shared `netcage-storage` subdir safely co-owned. So "two different Unix accounts run netcage on one host" is broken today.

This is not hypothetical. The sibling project anon-pi wants to run netcage AS A DIFFERENT Unix user (`netuser`), because that is how it scrubs the operator's real login name from the OTHER, un-fixed half of Leak 2: the `-v` bind-mount SOURCE paths. anon-pi needs to mount a home and project dirs into the jail, so it cannot follow ADR-0013's "mount from outside `$HOME`" advice; instead it runs the whole thing as `netuser`, so those `-v` source paths (and thus the tool's mountinfo) read `/home/netuser/...`, a purpose-built throwaway name, never the operator's real one. But the moment netcage runs as `netuser` alongside a login user who has already run it, the fixed `/var/tmp/netcage-storage` collides. The store relocatability that netcage needs here is not a privacy feature: it is the fix for the multi-user collision bug that the fixed path always had.

To be explicit about what this is NOT: this is NOT about hiding the store from other host PROCESSES (a login-user `find`/`grep` browsing the store). That is a different threat model (store-privacy against host processes) and is deliberately OUT of scope. The ONLY host-identity concern here is Leak 2, the in-jail TOOL reading the operator's login name from mountinfo. Everything below serves that and the multi-user correctness fix.

## Decision

**Make the graphroot DEFAULT uid-scoped, and PROMOTE `NETCAGE_GRAPHROOT` from a test-only seam to a supported OPTIONAL override.**

- **The default becomes `/var/tmp/netcage-storage-<uid>`**, where `<uid>` is the numeric UID of the user running netcage (`os.Getuid()`). Each Unix user therefore gets a distinct, self-owned store path with ZERO configuration, so netcage "just works" when run as any user (including anon-pi's `netuser`), and the multi-user collision bug is fixed. Every other property `/var/tmp` was chosen for is preserved: still under sticky `/var/tmp` (no root, no provisioning helper), still podman-created lazily and self-healing, still disk-backed (kept containers + image cache survive reboots), still cleared with `podman --root <path> system reset --force`.
- **A numeric UID in the path is acceptable for Leak 2, and this is the crux decision.** Leak 2 is about the operator's login NAME, not a uid. The in-jail tool reading mountinfo now sees `/var/tmp/netcage-storage-1000`, which carries a numeric uid but NOT the account name. A bare uid is not an identity leak in the sense Leak 2 cares about (it does not reveal WHO the operator is by name), and it is exactly what lets the default stay self-healing and configuration-free (an opaque per-user random subdir would force netcage to REMEMBER the path, losing idempotency + self-heal). ADR-0013's graphroot comment aspired to "no username AND no uid"; this consciously RELAXES that to "no username; a numeric uid is allowed", because name-free is what Leak 2 requires and uid-scoping is what multi-user correctness requires. The relaxation is deliberate and recorded, not an oversight.
- **`NETCAGE_GRAPHROOT` is promoted to a SUPPORTED optional override.** When set, netcage points its whole store at that path; when unset, it resolves the uid-scoped default. The multi-user case no longer NEEDS the override (the uid default handles it), so the knob is for callers who want an explicit path (e.g. a specific disk, a tmpfs, a deployment convention), not a requirement. Its code comment changes from "test-only / NOT a user-facing knob" to "supported optional override; also used by tests to isolate storage."

### Why uid-scope the default rather than leave it fixed + require the knob

Considered: keep the fixed `/var/tmp/netcage-storage` default and make `NETCAGE_GRAPHROOT` the (promoted) way any non-first user avoids the collision. REJECTED: it makes correct multi-user operation OPT-IN, so a second user who does not know to set the var hits the collision bug with a confusing permission error. The requirement is "netcage runs correctly as any Unix user"; a default that only works for ONE user per host and needs a knob for the rest does not meet it. Uid-scoping the default makes the common case correct by construction; the knob is then a genuine convenience override, not a workaround for a known bug.

### Why promote `NETCAGE_GRAPHROOT` itself rather than add a second knob

Considered: add a separate deployment-named override and keep `NETCAGE_GRAPHROOT` test-only. REJECTED. ADR-0013's single-store invariant is load-bearing (a split store makes `netcage ps`/`start` unable to find what `netcage run` created, a correctness break). Two env vars feeding one resolver forces a precedence rule, reintroducing exactly that split-store footgun. The integration TestMains in `internal/jail`, `internal/manage`, and `internal/verify` already point `NETCAGE_GRAPHROOT` at a scratch dir on every run: the "test-only" use IS the supported override, aimed at a temp dir. One knob, one resolver, one store keeps the invariant obvious; the "graphroot" name is podman's and acceptable (netcage is a thin podman client, and the operator is already reading about `--root`/`system reset`).

### Relationship to the ADR-0013 `--root` single-injection-seam (this is a resolver change, not new wiring)

The uid-scoped default and the promoted override both flow through the EXISTING seam. `podmanGlobalArgs` already prepends `--root <graphRoot()>` before every podman subcommand at `ExecRunner.Run`, and `jail.GraphRoot()` already re-resolves the same value for the one out-of-seam caller (the `forward` verb's socat child). This ADR changes ONLY what `graphRoot()` returns by default (a uid-scoped path instead of a fixed literal) and the intent of the env override. It adds NO new injection point, and MUST NOT: a per-builder `--root` would split the store.

### The `manage` verbs' `--root` refusal is UNCHANGED

The image-store write verbs (`build`/`pull`/`load`) REFUSE a user-supplied `--root` (`internal/manage/manage.go`, `refuseUserRoot`), because honouring it would re-split the single store. That refusal stays. The supported way to relocate is `NETCAGE_GRAPHROOT` (which moves the whole store through the one resolver), NOT a per-command `--root`. The refusal message currently names the fixed `/var/tmp/netcage-storage`; it should be updated to describe the store generically (the uid-scoped default or the override), so it does not print a path netcage no longer uses.

### `--runroot` stays at podman's default

ADR-0013 left `--runroot` at podman's default (co-locating it with `--root` and wiping both produced `acquiring lock ... file exists` refresh noise). That stays. podman's rootless default runroot is `$XDG_RUNTIME_DIR/containers`, already per-user (each user has their own `$XDG_RUNTIME_DIR`), so it does NOT share the multi-user collision the graphroot had. Only `--root` needs uid-scoping; `--runroot` is already per-user by construction.

## Considered options (rejected)

- **Keep the fixed default; require `NETCAGE_GRAPHROOT` for non-first users:** REJECTED (makes multi-user correctness opt-in; a second user hits a confusing collision, see above).
- **An opaque per-user random subdir (no uid in the path) as the default:** REJECTED. It would satisfy "no uid" but is NOT self-healing (netcage would have to persist and re-find the path), breaking the zero-provisioning property. A numeric uid is name-free enough for Leak 2 and keeps self-heal.
- **A second, deployment-named override alongside a test-only `NETCAGE_GRAPHROOT`:** REJECTED (two vars, one resolver, split-store precedence footgun; see above).
- **A per-command `--root` for relocation (loosening `manage`'s refusal):** REJECTED. It splits the store, the exact break `refuseUserRoot` and the single seam prevent.
- **Uid-scoping `--runroot` too:** UNNECESSARY. podman's default runroot is already per-user (`$XDG_RUNTIME_DIR`), and co-locating runroot with root reintroduced ADR-0013's lock-refresh noise. Only `--root` moves.
- **netcage doing anything about the `-v` bind-mount SOURCE paths:** OUT OF SCOPE. The `-v` half of Leak 2 is fixed by anon-pi RUNNING AS `netuser` (so the source paths read `/home/netuser/...`), which is anon-pi's choice, not netcage's. netcage's job is only to run CORRECTLY as whatever user it is launched as; it does not rewrite or validate `-v` sources (ADR-0013 already scoped that to the tool-launcher's concern).

## Consequences

- **Multi-user hosts work.** Two (or more) Unix users can run netcage on one host without colliding on the store; each gets `/var/tmp/netcage-storage-<their-uid>`. This is the bug fix.
- **The store path now varies by uid.** Docs, the `manage` `--root`-refusal message, and the store-cleanup instructions (`podman --root <path> system reset --force`, never `rm -rf` because the overlay `diff/` is subuid-owned) must describe the store as uid-scoped (or the override), not the old fixed literal. The README's existing `/var/tmp/netcage-storage` references need updating.
- **Leak 2 stays closed; a numeric uid is now in the path (accepted).** The in-jail tool sees a name-free, uid-bearing path. This is a conscious, recorded relaxation of ADR-0013's "no uid" aspiration, justified by Leak 2 being name-focused and by multi-user correctness needing the uid.
- **Does NOT touch forced egress or any other ADR-0013 closure.** Only the store path's default shape changes. The forced-egress leak-test (`verify`) and the hostname/NIC-name closures are unchanged.
- **Cross-repo:** this UNBLOCKS anon-pi running netcage AS `netuser` (no store collision with the login user). anon-pi still owns the `-v` half of Leak 2 (it runs as `netuser` and mounts `netuser`'s home/projects so the source paths carry no real login name); netcage does not fix `-v` sources.
- **This is a bug fix, not a new capability.** The fixed path was always multi-user-unsafe; uid-scoping corrects it. The promoted override is the small feature riding along.
