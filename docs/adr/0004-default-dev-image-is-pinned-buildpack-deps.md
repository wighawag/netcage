# Default dev image is a digest-pinned buildpack-deps, and a bare positional is a command

**Status:** accepted

When `netcage run` is given no positional image, it injects a DEFAULT dev image so pointing netcage at a repo folder is useful out of the box (prd `jailed-interactive-repo-run`, story 10). We pin that default to `docker.io/library/buildpack-deps:bookworm` by its immutable `@sha256:` digest (in `internal/devimage`, mirroring the redirector's digest-pin discipline from ADR-0001), and we resolve the image-vs-command ambiguity the default creates by treating a BARE first positional (no `/`, `:`, `@`, or `.`) as the command, not the image.

## Considered Options

- **buildpack-deps:bookworm (chosen).** The official broad Debian dev base: git, curl, wget, a C/C++ toolchain, python3, and the common development headers/libraries repos build against. It fits the untrusted-repo story (clone, install deps, build, run, all jailed) without netcage building or maintaining its own image, which the prd explicitly defers ("a bespoke/owned default dev image (Containerfile) ... is a later decision"). It is a multi-arch manifest list, so the digest resolves on amd64 and arm64. Cost: it is a large image, so its podman-gated integration test can be slow on a cold pull; acceptable because the test is gated (skips without podman) and the image caches after the first pull.
- **mcr.microsoft.com/devcontainers/universal (rejected).** Broader still (bakes in node/go/ruby/java runtimes), but multi-GB, which makes the podman-gated default-image run test impractical and the first real run painfully slow. The build toolchain + git + python in buildpack-deps covers the "set the repo up jailed" story; a repo that needs a specific runtime installs it in its jailed setup step, or the user passes a more specific positional image (which overrides the default).
- **A small base (alpine/busybox) (rejected).** Small and fast, but not a "broad, multi-language dev base": it lacks git and a build toolchain, so the untrusted-repo setup story it exists to serve would not work bare.

## Consequences

- **Image-vs-command disambiguation.** With no `--image` flag (podman-native positional grammar, ADR of the `podman-shaped-cli-flag-parsing` task) and a default in play, the first positional may be the image OR the command. We decided by FORM: a token that looks like an image reference (contains `/`, `:`, `@`, or `.`, e.g. `alpine:latest`, `docker.io/library/alpine`, `registry.example/x`) is the image; a bare command-shaped token (`bash`, `sh`, `python`) is the command with the default image injected, so `run -it bash` == default dev image running `bash`. An explicit `--` end-of-flags marker forces the next positional to be the image (`run -- alpine sh`), which is the escape hatch for a bare-token image name. This is reversible (it is a heuristic in one function, `resolveRunPositionals`) but recorded because a reader will otherwise wonder why bare `alpine` is treated as a command.
- **Repo-mount ergonomics use `/work`.** A `-v <repo>` with no container target defaults to `<repo>:/work`, and a mount at `/work` with no `-w` defaults the workdir to `/work`, so a repo dropped in is worked in without hand-writing `-w`. An explicit `-w` always overrides. `/work` is the mount target the prd's own stories use.
- **Re-pinning is deliberate.** The digest is auditable (provenance recorded in the `internal/devimage` package doc) and guarded by tests that reject a mutable tag, exactly like the redirector. Changing the default image or bumping its digest is an intentional edit, never an implicit `:latest` drift.
