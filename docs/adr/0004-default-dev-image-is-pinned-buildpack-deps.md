# Default dev image is a digest-pinned buildpack-deps, used only when no image is given

**Status:** accepted (supersedes the original bare-positional-is-a-command rule)

When `netcage run` is given no positional image, it injects a DEFAULT dev image so pointing netcage at a repo folder is useful out of the box (spec `jailed-interactive-repo-run`, story 10). We pin that default to `docker.io/library/buildpack-deps:bookworm` by its immutable `@sha256:` digest (in `internal/devimage`, mirroring the redirector's digest-pin discipline from ADR-0001). The default applies ONLY when there is no positional image at all; when a positional IS given, the first positional is always the image (podman-native grammar), with no image-vs-command guessing.

## Considered Options

- **buildpack-deps:bookworm (chosen).** The official broad Debian dev base: git, curl, wget, a C/C++ toolchain, python3, and the common development headers/libraries repos build against. It fits the untrusted-repo story (clone, install deps, build, run, all jailed) without netcage building or maintaining its own image, which the spec explicitly defers ("a bespoke/owned default dev image (Containerfile) ... is a later decision"). It is a multi-arch manifest list, so the digest resolves on amd64 and arm64. Cost: it is a large image, so its podman-gated integration test can be slow on a cold pull; acceptable because the test is gated (skips without podman) and the image caches after the first pull.
- **mcr.microsoft.com/devcontainers/universal (rejected).** Broader still (bakes in node/go/ruby/java runtimes), but multi-GB, which makes the podman-gated default-image run test impractical and the first real run painfully slow. The build toolchain + git + python in buildpack-deps covers the "set the repo up jailed" story; a repo that needs a specific runtime installs it in its jailed setup step, or the user passes a more specific positional image (which overrides the default).
- **A small base (alpine/busybox) (rejected).** Small and fast, but not a "broad, multi-language dev base": it lacks git and a build toolchain, so the untrusted-repo setup story it exists to serve would not work bare.

## Consequences

- **First positional is always the image (revised).** The original decision used a FORM heuristic: a token that looked like an image reference (contained `/`, `:`, `@`, or `.`) was the image, while a bare command-shaped token (`bash`, `sh`, `python`) was the command with the default image injected, so `run -it bash` meant default image running `bash` and a bare-token image needed a `--` marker (`run -- alpine sh`). That surprised users on the single most famous short image name (`alpine`), so it was reversed: **the first positional is now unconditionally the image**, exactly like `podman run IMAGE [CMD...]`, and the default image applies ONLY when no positional image is given. `run -it alpine sh` => image `alpine`, command `sh`, no marker needed. The `--` end-of-flags marker is still accepted but is no longer load-bearing. The tradeoff: `run -it bash` now means image `bash` (not default image + `bash`); to use the default image, give no positional (`run -it -v <repo>:/work`). This lives in one function, `resolveRunPositionals`.
- **Repo-mount ergonomics use `/work`.** A `-v <repo>` with no container target defaults to `<repo>:/work`, and a mount at `/work` with no `-w` defaults the workdir to `/work`, so a repo dropped in is worked in without hand-writing `-w`. An explicit `-w` always overrides. `/work` is the mount target the spec's own stories use.
- **Re-pinning is deliberate.** The digest is auditable (provenance recorded in the `internal/devimage` package doc) and guarded by tests that reject a mutable tag, exactly like the redirector. Changing the default image or bumping its digest is an intentional edit, never an implicit `:latest` drift.
